- **改动对象**：`deploy/tenant.env.example`（补全为权威模板）、`secrets/example.env` + `.env.example`（改软链）、`deploy/Dockerfile`、`deploy/deploy-remote.sh`、`README.md`。
- **类型**：部署基建 / 配置治理（无运行时代码、无 DB 变更）。
- **影响范围**：新增/更新租户 env 变量后，模板随每次 CI/CD 部署自动发到无源码目标机（test/prod3），运维在目标机编辑 `secrets/<tenant>.env` 时能看到最新完整变量清单。有源码机（prod2）走 git sync 不变。
- **触发原因**：fuduoduo（test 无源码机）scenarios 不生效，根因是其 `secrets/fuduoduo.env` 缺 `SCENARIO_COACH_ENABLED=true`（默认关闭）；深层问题是**无源码机拿不到任何 env 模板**——CI/CD 到无源码机唯一通道是 deploy 镜像（build context 仅 `deploy/`），此前只带 `compose-*.yml`+`deploy-remote.sh`，运维无从得知新增变量存在，形成「本地好、服务器坏」。
- **改动明细**：
  - **三份模板收敛为单一真源**：确立 `deploy/tenant.env.example` 为唯一权威模板（也是唯一能进 deploy 镜像、发到无源码机的位置——`deploy.yml` 用 `docker build -f deploy/Dockerfile deploy/`，build context 仅 `deploy/`）。`secrets/example.env`（`../deploy/tenant.env.example`）与根 `.env.example`（`deploy/tenant.env.example`）改为软链指向它，零漂移；三者仍纳入 git 追踪。`cp .env.example .env` / `cp secrets/example.env secrets/<t>.env` 因 `cp` 解引用软链，行为不变。
  - **模板补全**（键集 = `secrets/taishan.env`（最全实配）∪ 现有三份模板已有键）：补齐此前三份都缺的 `EMBEDDING_API_KEY/BASE_URL`、`DINGTALK_MEDIA_*`（图片/语音理解：`MEDIA_ENABLED/BASE_URL/API_KEY`+`VISION_MODEL`+`AUDIO_MODEL`）、`DINGTALK_AGENT_ID`、`HYBRID_RETRIEVAL`、`BOCHA_API_KEY`、`GUIDED_FLOWS_ENABLED`、`ONTOLOGY_VECTOR_FALLBACK`、`SCENARIO_COACH_*`。所有敏感值（`*_API_KEY`/`*_SECRET`/`NEO4J_PASSWORD`/`BOCHA_API_KEY`）用占位符；`DATABASE_URL`/`NEO4J_*` 注明由生成器注入 compose、仅本机 dev 需填。故意不入模板：`PROCESS_ROLE`（compose 按 api/stream/worker 角色注入）、`CHAT_MODEL`/`EMBEDDING_MODEL`（legacy 无前缀别名，模板用 taishan 生效约定的 `MODEL_CHAT_MODEL`/`MODEL_EMBEDDING_MODEL`）、`DINGTALK_MEDIA_API_KEY_ENV`（高级间接层，已直接提供 `DINGTALK_MEDIA_API_KEY`）。
  - **CI/CD 自动发布**：`deploy/Dockerfile` 加 `COPY tenant.env.example /deploy/`；`deploy-remote.sh` 在 compose up 前新增第 0 步——幂等 `cp -f /deploy/tenant.env.example $WORKSPACE/secrets/example.env`，只写保留名 `example.env`（`deploy-release.sh`/compose 都不部署它），绝不触碰真实 `<tenant>.env`。有源码机软链随 git 到位；无源码机由该步写成等价真实文件（`deploy/` 不在目标机，软链会悬空，故写真实副本）。
- **验证**：`bash -n deploy/deploy-remote.sh scripts/deploy-release.sh` 通过；`docker build -f deploy/Dockerfile deploy/` 成功且镜像内 `/deploy/tenant.env.example`（5894B）存在；落盘 dry-run 确认写出 `example.env` 且并存的假 `fuduoduo.env` md5 不变；`find secrets/ -type f ... ! -name example.env` 确认软链 example.env 不进 `deploy-release.sh` 租户菜单；`render-multitenant-deploy.py deploy/tenants.test.json` 仍成功；模板键 vs `taishan.env` 键差集为空、vs config `os.getenv` 基线差集仅上述 4 个故意排除项；grep 确认模板无真实密钥。
- **后续（本改动不含）**：fuduoduo 实际生效仍需运维在 test 机 `secrets/fuduoduo.env` 加 `SCENARIO_COACH_ENABLED=true` 并重启 `fuduoduo-stream/api/worker`，且确认 test 镜像为今日含 scenario 代码的新构建。

### 类型

### 影响范围

- **Online Graph 持久化**：所有在线对话轮次状态现在存储在 PostgreSQL 中，跨进程重启和 worker 变更保持一致
- **Stable thread ID**：线程标识为 `online:<tenant>:<agent>:<channel>:<user>`（无日期组件）
- **Turn-scoped reset**：每轮显式重置瞬时字段
- **Advisory lock 序列化**：相同线程的轮次按进程序列化
- **Bounded topic restore**：关闭 ≤30min 的主题自动恢复
- **统一标准路径/流式路径**：两个路径共享 prepare_online_turn
- **DingTalk 重复投递静默处理**：重复事件零副作用
- **Health endpoint**：`/ready` 包含 checkpoint ready 状态

### 改动明细

1. **PostgreSQL checkpoint runtime（Task 1）**：`checkpoint_runtime.py`（Fail-closed 语义、无 InMemorySaver 回退）
2. **Lifecycle wiring（Task 2）**：FastAPI/Stream/Worker 角色初始化/关闭 checkpoint runtime，`/ready` 补充 checkpoint 状态
3. **Stable thread + turn envelope（Task 3）**：`build_online_thread_id` 去除日期组件，`TURN_SCOPED_DEFAULTS` + `build_online_turn_input`
4. **Advisory lock + dedup（Task 4）**：`online_turn_lock.py`（blake2b signed bigint + pg_advisory_xact_lock），processor duplicate no-op
5. **Bounded topic restore（Task 5）**：`topic_restore.py` / `TopicRestoreDecision` / restore anchor + 多候选话题澄清流程
6. **Unified standard/streaming（Task 6）**：`prepare_online_turn` + `PreparedOnlineTurn` + graph_stream 迁移 + processor DingTalkTurnResult + reset 状态机
7. **Router evaluation repair（Task 7）**：async evaluator + 20 clarification resolution cases
8. **Multi-turn scenarios + gate（Task 8）**：24 JSONL scenarios + fixture mode eval + deterministic gate
9. **Release gates + runbook（Task 9）**：`run_short_term_memory_gate.sh` + `docs/runbooks/short-term-memory.md`

### 原因

生产环境 Online Graph 依赖进程内 InMemorySaver（重启丢失所有对话状态），Thread ID 含日期（凌晨重连导致状态丢失），Stream/HTTP 路径使用不同的 Thread ID 和 Graph 配置。

此次升级将 Online Graph 改造为基于 PostgreSQL 的持久化、稳定架构。
