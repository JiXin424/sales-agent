  - **`promoted_regressions`（`PromotedRegression`）**：从 trace 提升的匿名 regression 场景。列：`id/tenant_id/source_trace_id/scenario_json(JSON)/status(default 'draft', draft→reviewed→committed)/anonymized(default true)/review_notes/created_at/updated_at`。索引：`ix_promoted_regressions_tenant_id`。
  - 模型在 `models/__init__.py` 第 81 行注册（`from .memory_eval import MemoryEvalTraceRecord, PromotedRegression`），并加入 `__all__`，供 Alembic `env.py` 的 metadata 发现。
  - Migration `0015_memory_eval_operations`：`down_revision = "0014_user_memory_profiles"`，仅 `create_table` 两张表（不混 add_column），遵循 0014 命名规范（`Text` 列、显式 `PrimaryKeyConstraint`、独立 `create_index`、时间戳列用 `Text + server_default=func.now()` 与 `TimestampMixin` 一致）。
- **原因**：Task 18 为 23 任务计划的存储层基础；后续 online-sample（Task 19）、promote-trace（Task 21）等模式需要这两张表落库。
- **验证**：集成测试 `tests/integration/test_memory_eval_models.py`（2 项）经 `db_session` fixture（`Base.metadata.create_all`）全绿，证明模型正确（primary gate）。Migration 单独验证：测试库 `create_all` 建全表 → drop 两张新表 → `alembic stamp 0014_user_memory_profiles` → `alembic upgrade head` 仅应用 0015 成功，表结构与模型一致（列类型/默认/索引核对通过），测试库到 head `0015_memory_eval_operations`。

## Eval 租户绑定修复：graph-multiturn / dingtalk-staging 独立 CLI 跑通 chat-path（Spec 4 §3.2,§3.4）

- **改动对象**：`eval/memory_eval/runner.py`（`_bind_test_database` / `_restore_database`）、`tests/integration/test_memory_eval_graph_multiturn.py`、`tests/integration/test_memory_eval_dingtalk_staging.py`。
- **类型**：bug 修复（评估 harness 独立运行缺陷）。
- **影响范围**：仅评估 harness 离线入口；**不改动在线 Graph 请求路径**，生产流量零影响。
- **改动明细**：
  - **根因**：`graph-multiturn` / `dingtalk-staging` 独立 CLI 跑测试库时，chat-path 场景路由到 Graph 的 `resolve_tenant` 节点 → `TenantResolver.resolve` → `get_tenant_runtime().check_tenant_match(tenant_id)`；dedicated 模式下 runtime 租户取自 `TENANT_ID` 环境变量，与 `args.tenant_id`（`eval-tenant`）不等，抛 `TenantMismatchError`，场景失败。既有集成测试只用 explicit-remember 场景（走 `memory_command`，绕过 `resolve_tenant`），未覆盖此路径。
  - **修复**：`_bind_test_database(test_db, tenant_id)` 在绑定测试库时同步把 `os.environ["TENANT_ID"]` 设为 eval 租户并置空 `tenant_runtime._runtime` 单例（`load_dotenv` 不覆盖已存在 env，故此处设置胜出）；`_restore_database` 恢复原 `TENANT_ID` 并再次置空单例，避免泄漏。`run_graph_multiturn` / `run_dingtalk_staging` 两处调用点传入 `args.tenant_id`。`_seed_eval_fixtures` 已种子该租户行，故 `TenantResolver.resolve` 可命中。
  - **测试加固**：两个集成测试的内联数据集各新增一个 chat-path 场景（如「帮我看看福多多产品怎么讲」，`turn_relation:"new"`），端到端触发 `tenant_resolve`；`test_memory_eval_graph_multiturn.py` 新增 `test_graph_multiturn_chat_path_runs_past_tenant_resolve` 直接断言 chat turn `response_kind != "duplicate"` 且无 error（因聚合报告不序列化逐轮 error，ScenarioRunner 会吞掉异常）。dingtalk-staging 的 turn 循环无 try/except，故 tenant 崩溃会直接传播使测试失败——加 `total_scenarios == 2` 断言即足够守卫；chat 场景 event_id 用 uuid 后缀避免 `memory_outbox` 唯一约束在持久测试库上重跑碰撞。
- **原因**：补齐 Spec 4 头条交付物「一条命令跑 ≥40 场景」——修复前 chat-path 全部崩在 tenant_resolve，报告 hollow。
- **验证**：独立 CLI `graph-multiturn` 跑完 42 场景无 `TenantMismatchError`，非 hollow（`topic_leakage_rate 13/52`、`explicit_operation_accuracy 10/26`）；门禁 Phase B 仅因质量门禁失败（零 tenant 相关失败）；`pytest tests/unit/eval tests/integration/test_memory_eval_graph_multiturn.py tests/integration/test_memory_eval_dingtalk_staging.py -q` → 82 passed；模拟未绑定租户确认 `run.error` 为 TenantMismatchError（回归守卫有效）。
