

# 2026-07-09 升级日志

## 钉钉语音/图片支持流式互动卡片

- **改动对象**：`src/sales_agent/integrations/dingtalk/stream_client.py`、`src/sales_agent/integrations/dingtalk/graph_stream.py`；新增测试 `tests/unit/dingtalk/test_stream_client.py`。
- **类型**：bug 修复 + 功能补全（让语音/图片与文字享受同一流式卡片路径）。
- **影响范围**：仅钉钉 **Stream** 模式（生产主入口），且仅在 `dingtalk.streaming_enabled=true`（默认关）+ `dingtalk.media_enabled=true`（默认开）时生效。HTTP 回调模式、关闭流式的实例行为零变化。无 DB 变更、无 Alembic migration、无配置 schema 变化。
- **改动明细**：
  - **根因**：`stream_client.py` 的流式门禁硬编码 `message_type == "text"`，语音(`voice`/`audio`)/图片(`picture`/`image`/`richText`)被直接挡进非流式 `handle_dingtalk_event`；且流式分支 `_handle_streaming` 虽接收 `media_download_codes`/`raw_event` 参数却从未使用，也没调 `DingTalkMediaAdapter`——媒体转不出文字，是没接完的半成品。
  - **改动①门禁**：抽出模块级纯函数 `_should_stream(config, message_type, text_content)`——文字（非快速命令）或受支持媒体（`media_enabled` 开）才进流式，其余不变。`process()` 内 `use_streaming` 改调它。
  - **改动②透传类型**：`_handle_streaming` 签名 + 调用处补 `message_type` 参数（此前未传）。
  - **改动③接媒体适配**：新增模块级 `_resolve_streaming_message(...)`——文字直接透传不开卡；媒体先开「正在识别…」过渡卡片（阻塞 ASR/视觉期间给用户反馈），再 `DingTalkMediaAdapter.to_agent_text` 转写，返回 `(text, card_id)`；转写失败在**同一张卡** `streaming_finalize` 友好提示（`reply_fn` 仅作卡片 finalize 也失败时的兜底）。`_handle_streaming` 调用它，`message_text is None` 时直接 return 不进 Graph。
  - **改动④复用卡片**：`handle_dingtalk_stream_via_graph` 加可选 `card_id: str | None = None`——传入则复用过渡卡、不新建「分析中…」卡，实现「正在识别 → 分析中 → 回答」单卡 UX；不传则维持原行为（文字路径零回归）。
  - **改动⑤修兜底**：`_handle_streaming` 在 `CardSender` 不可用降级时，原先硬写 `message_type="text"` 且丢弃 `media_download_codes`/`raw_event`，导致媒体消息无法转写；改为透传真实 `message_type` 与媒体字段，落到 `handle_dingtalk_event` 的媒体分支兜底。
  - **未夹带**：token 级流式隐患（生成节点 `generate_node` 经 `execute_agent` → `chat_model.generate()` 为非流式调用，真正的 `stream_generate` 全仓无调用方；`graph_stream.py` 中 `GRAPH_CHUNK`/`GRAPH_DONE`/`CARD_FINALIZE` 满屏 `logger.warning` 调试日志印证团队已在排查）属独立问题，本次不动，留作后续任务。
- **原因**：用户反馈「钉钉发语音/图片不能流式，只有文字能流式」。这是显式架构门禁 + 流式路径未接媒体的组合问题，非偶发 bug。目标是让媒体走与文字完全相同的流式路径（parity），而不是另起一套。
