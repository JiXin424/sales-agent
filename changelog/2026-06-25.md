# 2026-06-25 升级日志

## Neo4j Ontology Knowledge Engine（本体知识引擎）

### 概述
新增 Neo4j 本体知识引擎（`KNOWLEDGE_ENGINE=ontology_neo4j`），用可审计的
Entity / Fact / Evidence 知识图谱替代传统 Markdown chunk RAG。首版采用图检索 + 保守
向量回退，支持入库可视化与高风险事实人工复核，并保持 chat 输出与现有 `summary` /
`sections` 响应格式兼容。传统 chunk RAG 路径保持不变，可通过环境变量切换。

---

### 改动明细

#### 1. Ontology 核心包
- **改动对象**：`sales_agent.ontology` 包（schemas / canonicalizer / neo4j_client / schema /
  repository / extractor / conflict / ingestion_service / retrieval_service / answer_service）
- **类型**：新功能
- **影响范围**：知识引擎核心逻辑
- **改动明细**：
  - `schemas` / `canonicalizer`：本体 schema 定义与实体规范化。
  - `neo4j_client`：Neo4j 异步连接管理。

## Console 前端 dev 连接后端端口修复

### 概述
修复 `console` 下 `npm run dev` 报「当前实例尚未配置 Agent」+
`net::ERR_CONNECTION_REFUSED http://localhost:800X/instance/agent` 的问题。两层根因：

1. **端口错配**：前端 dev 指向宿主机 8001，而本机 8001 被**另一个项目** `ai-coach-fastapi`
   占用（404，非 sales-agent）；sales-agent 后端实际按 `deploy/tenants.json`（`api_port: 8003`）
   跑在 **8003**（`curl :8003/instance/agent` → 200）。
2. **绝对地址对远程浏览器无效**：把 env 改成 `http://localhost:8003` 后，服务器侧 curl 仍 200，
   但浏览器里的 `localhost:8003` 指向**用户自己的机器**（浏览器与服务器不同机）→ 依旧 refused。
   说明 dev 不能用 `http://localhost:<port>` 直连，必须让浏览器走**同源相对路径**，由 vite 在
   **服务器侧**代理到后端。

前端 fetch 失败后 `App.tsx` 的 `InstanceEntry` 一律用「未配置 Agent」兜底，造成误诊
（Agent 实际存在且 active）。

### 改动明细

#### 1. vite dev 代理（核心修复）
- **改动对象**：`console/vite.config.ts`（`server.proxy['/api']`）
- **类型**：配置 / bug 修复
- **影响范围**：本地前端开发
- **改动明细**：代理 target 由 `http://localhost:8000` 改为 `http://localhost:8003`（与本机后端一致），
  保留 `rewrite` 去掉 `/api` 前缀；补注释说明为何用代理（远程浏览器同机直连不可达）。
- **原因**：恢复项目本意的 dev 架构——浏览器同源 `/api/*`，由 vite 转发到后端，无论浏览器是否
  与服务器同机都成立。

#### 2. 前端 dev API 基址改同源前缀
- **改动对象**：`console/.env.development`
- **类型**：配置 / bug 修复
- **影响范围**：本地前端开发
- **改动明细**：`VITE_API_BASE_URL` 由绝对地址 `http://localhost:8001`（后曾试 `:8003`）改为
  同源前缀 `/api`；配合 #1，浏览器请求 `/api/instance/agent` → vite 转发到 `:8003/instance/agent`。
  生产构建仍同源（`.env.production` 留空），行为不变。
- **原因**：消除对 `localhost:<port>` 的依赖，根治远程浏览器 refused。

#### 3. 手写 dev compose（未改，说明）
- **改动对象**：`docker-compose.yml`（`taishan-api`，profile `taishan-split`）保持 `8001:8000`。
- **说明**：本机 8001 实际被 ai-coach 占用，运行中的 sales-agent 后端在 **8003**
  （由 `deploy/tenants.json` 生成的 `docker-compose.generated.yml` 决定）。本次**不改** compose：
  前端通过 vite 代理直连运行中的 8003，不依赖手写 compose 的端口。手写 compose 端口口径
  与生成式部署不一致是既有现象（如在本机用 `--profile taishan-split up` 会与 ai-coach 抢 8001），
  属独立问题，单独跟进。

### 验证
- 启动 `npm run dev` 后，`curl http://localhost:<vite端口>/api/instance/agent`（即浏览器将发出的
  同源请求）→ **200**，返回 taishan Agent（id/tenant_id/status=active）——证明代理对任意能访问
  vite 的浏览器都成立。
- 从转换后的 `src/api/client.ts` 核实 vite 注入 `VITE_API_BASE_URL="/api"`。
- `console` 测试套件 `npm test` → 10/10 通过。

### 影响边界 / 不改动
- **生产配置不动**：`deploy/tenants.prod3.json`（`api_port: 8001`）、
  `scripts/deploy-image-retag.sh`（默认 8001）、prod 部署文档均为**另一台机器**，
  其上 8001 正确，不在本次范围。
- `console/.env.production` 仍为同源（空），生产构建行为不变。

### 未完成 / 后续
- `console/src/App.tsx` 的 `InstanceEntry` 当前对**任意** fetch 失败（连接拒绝/404/超时）
  一律显示「未配置 Agent」，易把连通性问题误判为业务缺数据。建议区分「网络错误」与
  「确无 Agent」两种文案（本次未改，避免与端口修复捆绑；可单独跟进）。

## Neo4j 容器化接入 + 向量回退来源修复（2026-06-25）

### 改动对象
Neo4j 本地开发/运行容器、本体引擎真实集成测试、检索服务向量回退逻辑。

### 类型
feat（基础设施 + 测试） / fix（检索 bug）

### 影响范围
`docker-compose.yml`、`docs/ontology-neo4j-ops.md`、`tests/integration/test_ontology_neo4j_live.py`（新增）、`src/sales_agent/ontology/repository.py`、`src/sales_agent/ontology/retrieval_service.py`、`tests/unit/ontology/test_retrieval_service.py`。

### 改动明细
- **docker-compose**：新增 `neo4j` 服务（profile `ontology`，镜像 `docker.1ms.run/neo4j:5`，端口 7474/7687，卷 `neo4jdata`，`cypher-shell` healthcheck，测试账密 `neo4j/neo4jtest123`）。默认不启动，需 `docker compose --profile ontology up -d neo4j` 显式启用，不影响 legacy 部署。
- **Live 集成测试**：`tests/integration/test_ontology_neo4j_live.py`（gated by `NEO4J_LIVE_TEST`），用真实 Neo4j + fake LLM 验证 schema 启动初始化（约束 + 向量索引）、ingest→检索→回答端到端、保守向量回退。补上此前"repository Cypher 只被 fake 覆盖"的盲区。
- **fix 向量回退**：原向量回退只返回命中实体，不收集 facts/evidence/documents，导致非实体名子串的提问（走向量回退）回答无任何来源。`vector_query_statement` 改为同时返回 facts/evidence/documents；`OntologyRetrievalService` 向量回退分支收集它们。并在 `test_retrieval_service.py` 加单元测试保护。

### 原因
- 计划原本把 Neo4j 当外部服务、未提供本地启动方案，无法端到端真实验证；容器化后可一键起。
- live 测试暴露了向量回退无来源的真实 bug（fake 测试因 query_vector 只返回 `e` 而漏检）。

### 验证
- `docker compose --profile ontology up -d neo4j` → 容器 healthy。
- `NEO4J_LIVE_TEST=1 pytest tests/integration/test_ontology_neo4j_live.py` → 3/3 通过。
- 全量回归（ontology 单元 + live + task_router/risk_checker/processing_notice/coach）→ 109 passed。

### Evidence 补 tenant_id（已修）
- `Evidence` 节点此前不带 `tenant_id`（Entity/Fact/SourceDocument 都带），按租户清理时会残留孤儿。已在 `create_fact_statement` 给 Evidence 补 `tenant_id`，使按租户清理覆盖 Evidence、保持租户隔离一致性。真实 Neo4j live 测试验证：teardown 后无孤儿残留。
