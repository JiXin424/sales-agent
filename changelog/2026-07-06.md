## Ontology 检索内联化 — 修复竞品问答绕过主生成 prompt 的 bug

### 背景
用户问机器人「福多多的竞品有哪些」，机器人直接把 ontology 图谱检索结果原样端上来（"推断聚优和东福是竞品"），没有结合 RAG 综合、没有 markdown 格式、没有销售话术。根因：graph 新路径（钉钉 Stream）走 ontology 时，子图自己用一道简陋 prompt 生成答案并设 `skip_generation=True`，**完全绕过了主 `generate_node`**——而 PromptRegistry（租户定制 system/task prompt）只在 `generate_node` 生效。

### 改动明细
| 改动对象 | 类型 | 影响范围 | 改动明细 | 原因 |
|---------|------|---------|---------|------|
| `graph/nodes/retrieval.py` | 重构 | 检索节点 | 删除 `_get_ontology_subgraph`（lru_cache 包装的子图编译）和 `_retrieve_via_ontology` 的子图调用；改为直接顺序调用步骤函数 `extract_terms_node → graph_query_node → [vector_fallback] → compact_evidence_node`，构建 `ontology_context_text`，**永不设 `skip_generation=True`** | 消除 ontology 自己生成答案的旁路，让证据回流主生成节点 |
| `graph/retrieval/ontology_graph.py` | 重构 | ontology 步骤 | 删除 `generate_answer_node`、`build_ontology_retrieval_graph`、`should_vector_fallback`、`_ONTOLOGY_RESPONSE_PROMPT`；保留 4 个检索步骤函数（签名从 `OntologyRetrievalState` 改为 `dict`）；`vector_fallback_node` 的 limit 改读 `settings.ontology.vector_fallback_top_k` | ontology 不再自己出答案，只产证据；步骤函数直接被 retrieve_node 调用 |
| `graph/nodes/generation.py` | 修改 | 生成节点 | `execute_agent` 调用新增 `ontology_context=` 参数（从 state 读 `ontology_context_text`）；更新 `skip_generation` 注释/reason 文案 | ontology 证据注入主生成管道，PromptRegistry 的 system/task prompt 对 ontology 路径生效 |
| `graph/state.py` | 新增字段 | 图状态 | `ChatGraphState` 新增 `ontology_context_text: str` | 承载 ontology 证据文本从 retrieve 流到 generate |

## 数据库 schema 漂移修复（prod3 钉钉 stream crash 根因）

### 现象
prod3 的 `taishanyanshi` / `songbai` 两个租户的 `-stream` 容器虽 `Up`，但每次处理钉钉消息都报 `InFailedSQLTransactionError: current transaction is aborted`，graph 新路径完全不可用；同镜像（`eb889df`）在 prod2（taishan 开发机）正常。

### 根因（两层）
1. **直接原因**：prod3 的 `conversation_messages` 表缺 `topic_id` 列（`feat/topicandmemory` / migration `0011` 引入）。graph 新路径 `context_load.py:59` 的 ORM 查询带 `topic_id` → PostgreSQL `UndefinedColumn` → 整个事务进入 aborted 状态 → 后续所有 SQL 报 `InFailedSQLTransactionError`。
2. **根因（init_db 幽灵漂移）**：`core/database.py` 的 `init_db()` 原顺序是 `create_all()` → `alembic upgrade head()`。当一个 migration 同时含 `create_table`（新表）+ `add_column`（老表加列）时——典型如 `0011`（新建 `conversation_topics` + 给 `conversation_messages` 加 `topic_id`）——`create_all` 会抢先建好 `conversation_topics`（ORM model 已有），导致 `alembic upgrade` 撞 `DuplicateTableError`，触发 `_run_auto_migrations` 的 `stamp head` 兜底，**整体跳过同一 migration 里剩余的 `add_column`**。结果：`alembic_version` 标到 head，但老表的新列永远没加上（「版本号到位、schema 没到位」的幽灵漂移）。prod2 是开发机、库历史不同列已在；prod3 是 image-deploy 生产库，被这个 bug 漂移了。

### 改动
| 改动对象 | 类型 | 影响范围 | 改动明细 | 原因 |
|---------|------|---------|---------|------|
| `migrations/versions/0012_backfill_skipped_columns.py` | 新增 | DB schema | 幂等 backfill：`ADD COLUMN IF NOT EXISTS` / `CREATE INDEX IF NOT EXISTS` 补齐 prod3 缺失的全部列与索引（`conversation_messages.topic_id` + `document_chunks` 4 列 + `eval_cases` 14 列 + `eval_run_results` 12 列 + `eval_runs` 11 列 + `eval_suites` 7 列 + `retrieval_profiles` 5 列 + 6 个索引）。列定义与 prod2 权威库一致；schema 已完整环境执行为 no-op | 补回被 stamp 兜底跳过的历史 DDL，拉平 prod3 漂移；全是 `IF NOT EXISTS`、无 `create_table`，不会被 stamp 兜底再次跳过 |
| `core/database.py` `init_db()` | 修改 | 全环境启动 | 调换顺序：先 `_run_auto_migrations`（upgrade head）→ 再 `create_all`。upgrade 时 create_all 尚未预建 migration 表，`create_table` 不撞 `DuplicateTableError`，`add_column` 正常执行；create_all 随后幂等补建 model-only 表 | 修根因（前向）：未来「create_table + add_column」同 migration 不再漂移；新库路径（stamp head → create_all 全量）经验证不回归 |
| `core/database.py` `_run_auto_migrations()` | 修改 | 全环境启动 | 重写 docstring 说明新顺序契约；upgrade 失败兜底日志从「静默 stamp head」改为打印完整异常 + 警告「失败 revision 的 add_column 被跳过、必须 backfill」 | 让幽灵漂移在日志里可被发现，而非等 stream crash |

### 影响范围 / 局限
- **prod3**：`0012` 部署后任一容器启动 init_db → upgrade `0011→0012` 补齐 `topic_id` 等 → graph 新路径恢复。
- **prod2 / test**：`0012` 为 no-op（列已存在），新顺序对干净完整库无副作用（已验证）。
- **局限（诚实记录）**：若某表在「历史上」已被 create_all 预建（model 早于其 create_table migration 引入），upgrade 重跑该 migration 仍会撞 → 走兜底 stamp。彻底防范需「新 model 表随即配 create_table migration」+ CI schema 一致性校验（见下）。

### CI/CD 如何纳入 DB 变更（流程建议）
当前 CI（`.gitea/workflows/deploy.yml`）只构建+分发镜像，DB schema 靠容器启动时 `init_db` 自动迁移——这把 migration 执行隐式塞进运行时，漂移只能靠业务 crash 暴露。建议（按优先级）：
1. **部署后 schema 一致性校验（最高优先）**：fan-out 部署完成后，在每个目标机跑一个检查——对比 `alembic_version` 与关键表的实际列（如 `conversation_messages.topic_id` 是否存在），不一致即 CI 失败报警。本次 prod3 就是缺这层校验，漂移了 4 个 commit 才被发现。
2. **migration 预演（pre-deploy dry-run）**：CI 构建阶段对新 migration 在临时 pgvector 库跑一次 `alembic upgrade` + 语法校验，避免有语法/逻辑错误的 migration 进镜像。
3. **CI 顺序约束**：含 DB migration 的 commit，先让一个 `api` 容器完成 `init_db`（upgrade）再起 `stream`/`worker`，避免 stream 先起来查到旧 schema。
4. **规范**：新 ORM 表必须**同一 PR** 配 `create_table` migration（不要 model 先进、migration 后补，否则 create_all 会预建、埋下漂移）。

### 验证
- prod3 事务 dry-run：53 条 `ALTER` + 6 条 `CREATE INDEX` 全部成功；`topic_id` 事务内可见、ROLLBACK 后归零（未污染生产数据）。
- 临时 pgvector 容器跑 B1 版 `init_db` 新库路径：`alembic_version=0012`、`conversation_messages.topic_id` / `conversation_topics` 表 / `document_chunks` 新列 / 索引齐全、71 张表完整。
- 部署后复查 `docker logs <tenant>-stream`：无 `InFailedSQLTransactionError`，graph 正常产出答案。

## CI 部署后 DB schema 一致性校验（落地）

### 背景
上节「数据库 schema 漂移修复」的 CI/CD 流程建议第 1 条（最高优先）：fan-out 部署后对比 `alembic_version` 与实际列，漂移即 CI 失败。prod3 这次正是缺这层校验，漂移了 4 个 commit 才被 stream crash 暴露。本节落地该建议——把幽灵漂移拦在部署阶段，不再等业务 crash。

### 改动明细
| 改动对象 | 类型 | 影响范围 | 改动明细 | 原因 |
|---------|------|---------|---------|------|
| `scripts/check_schema_consistency.py` | 新增 | DB 校验 | 连 DB 查 `information_schema`，对比 ORM `Base.metadata` 的所有表/列是否真实存在；缺即 exit 1。校验基准用 ORM model（应用真正会查的列），而非解析 migration 文件——天然忽略「migration 加了但 model 没同步」的列（如 `retrieval_profiles.entity_limit`，应用从 config 读），只报「model 有、DB 没有」的真实漂移 | 捕获 stamp-head 幽灵漂移：`alembic_version=head` 但列没落地，version 表查不出，只能查实际列 |
| `scripts/post-deploy-schema-check.sh` | 新增 | 部署校验 | helper：找 running 的 `-api` 容器 → 等 `/health` 200（= FastAPI lifespan 完成 = `init_db()` 已跑完）→ `docker exec` 跑校验脚本 | 时机关键：`init_db` 在 lifespan startup 同步 await，`/health` 200 是其完成的可靠信号；先等就绪再校验，避免迁移未跑完误报 |
| `deploy/deploy-remote.sh` | 修改 | image-deploy 链路（prod3/test） | step 3 从「只 grep `-api` running 就 exit 0」改为「等 api running → 等 `/health` 200 → exec schema 校验 → 通过才 exit 0」。校验失败 exit 1 中止部署 | 修 CLAUDE.md 指出的盲点：旧版只查容器 running 不查 app 就绪/schema，stream 漂移 crash 漏网；image-deploy 链路必须自带校验（deploy 镜像无 scripts/，故 inline 同 helper 逻辑） |
| `scripts/deploy-release.sh` | 修改 | deploy-release 链路（prod2） | step 11（health checks）后新增 step 11b，调 `post-deploy-schema-check.sh` | 本机部署链路补上同款校验，两条链路对齐 |
| `scripts/ci-fanout.sh` | 修改 | CI fan-out 失败传播 | 6 处部署失败的 `\|\| echo "⚠️ 继续下一台"` 改为 `\|\| { echo; FAILED=1; }`，循环末尾 `FAILED=1` 时 `exit 1`；运维脚本同步（非关键步骤）保留 `\|\| echo` | 修 `\|\| echo` 吞 exit code 缺陷：否则 deploy-remote.sh 校验失败 exit 1 被吞，CI job 仍 success（只日志 ❌），「漂移即 CI 失败」承诺失效。详见 lessons #29 |

### 设计要点
- **校验基准 = ORM `Base.metadata`**，不是 migration 文件。`Base.metadata` 是应用真正会 `SELECT` 的列集合；migration 文件会含历史遗留列（model 已删但 migration 留着）或 model 未同步列，用作基准会误报。单向检查「model ⊆ DB」，DB 多余的表/列不报（无害）。
- **时机 = `/health` 200**。`init_db()` 在 `lifespan` startup 同步 await（`main.py:39`），路由就绪即迁移完成。`/health` 只返回 `{"status":"ok"}` 不依赖 DB，但 FastAPI 在 lifespan 完成后才挂载路由，故 `/health` 200 是 `init_db` 完成的可靠信号——成功或 stamp-head 兜底都算「完成」，schema 状态稳定可校验。
- **连 DB 失败 → exit 0**（交 health check 兜底）。校验本身不把「DB 未就绪」误判为漂移；只有连得上且列缺失才 exit 1。
- **共享 DB，校验一次**。多租户共享一个 DB（tenant_id 区分），挑任一 `-api` 容器 exec 即覆盖全部租户。
- **失败传播到 CI job**。`ci-fanout.sh` 原用 `|| echo` 容错会吞掉 deploy-remote.sh 的 exit 1（校验失败时 CI 仍 success），改为收集 `FAILED` 标志末尾 `exit 1`，让 schema 校验失败真正红 CI（运维脚本同步等非关键步骤仍保留 `|| echo` 不阻断）。

### 验证
- 校验脚本三场景：① 无 `DATABASE_URL` dry-run → exit 0；② prod2 完整库 → exit 0（70 表/1020 列全在）；③ 临时库 drop `conversation_messages.topic_id` → exit 1 精确报该列（且 `alembic_version=None` 仍抓到，证明不依赖 version 表、stamp 撒谎也抓得住）。临时库已清理。
- helper 端到端：prod2 现有容器（临时 cp 脚本进旧镜像容器）→ 找到 `sales-agent-taishan-api` → `/health` 1×2s 就绪 → exec 校验通过。新镜像 build 后脚本随 `COPY scripts/` 进镜像，无需手动 cp。
- 两个部署脚本 `bash -n` 语法检查通过。
- `ci-fanout.sh` `bash -n` 语法通过；6 处 `FAILED=1` 收集 + 末尾 `exit 1` 逻辑审查正确，运维脚本同步（第 105 行）保留 `|| echo`。push 后 CI 跑正常路径（schema 校验通过、`FAILED=0`、CI success）验证未破坏正常部署；失败路径（校验失败→`exit 1`→CI 红）由逻辑保证。
