## Graph Debug 新增只读「checkpoint 时间旅行」（时间轴）

- **改动对象**：`src/sales_agent/api/routes/graph_debug.py`、`console/src/api/graphDebug.ts`、`console/src/api/types.ts`、`console/src/pages/Agents/GraphDebugPage.tsx`、新增 `console/src/pages/Agents/JsonNode.tsx` + `GraphDebugPage.css`。
- **类型**：功能升级（开发者调试工具，A1 只读范围）。
- **影响范围**：仅 Graph Debug 调试路径（`/agents/{id}/graph-debug/*`）；不触碰生产 HTTP `/agent/chat`、钉钉流式路径；不动 DB schema（checkpoint 表由 LangGraph `AsyncPostgresSaver` 自管，无 Alembic migration）。
- **改动明细**：
  - 后端 `/run` 改用 `await get_checkpointer()`（`AsyncPostgresSaver` 单例；无 `DATABASE_URL` 时回退 `InMemorySaver`）编译图，使每个节点 boundary 写 checkpoint；`thread_id = f"debug:{uuid4()}"`，`conversation_id` 同步；首个 SSE `started` 事件回传 `thread_id`。
  - 新增 `GET /agents/{agent_id}/graph-debug/threads/{thread_id}/checkpoints`：返回 `{"checkpoints":[{checkpoint_id, step, node, ts, next}]}`（按 step 升序）。
  - 新增 `GET .../threads/{thread_id}/checkpoints/{checkpoint_id}/state`：返回 `{values, next}`。
  - 两端点强制 `thread_id` 以 `debug:` 开头，否则 403（隔离生产会话 state）。
  - 前端：`getCheckpoints` / `getCheckpointState` API；`GraphDebugPage` 解析 `started.thread_id`，run 完拉 checkpoint 链渲染时间轴（Ant Design `Steps`），点步用自写 `<JsonNode>`（**零新依赖**）展示完整 state；`localStorage["graph-debug:runs"]` 存最近 20 条 run，顶部下拉切换重载；Tabs 区分「实时 trace」/「Checkpoint 时间轴」。
- **原因**：调试 routing 走偏 / retrieval 拉错 / risk 误杀时，原 graph-debug 跑完即忘（`builder.compile()` 未接 checkpointer），无法回看每步 state。LangGraph 原生支持 `aget_state_history` 回溯，基础设施（`AsyncPostgresSaver`）已就绪，本次只做接线 + 前端可视化。选 Postgres（非内存）以兼容多 worker；`debug:` 前缀隔离生产数据。
- **验证**：进程内验证（InMemorySaver）通过——端点形状 `{checkpoints:[...]}`、字段映射（step/checkpoint_id/ts/next/node）、403、state 详情均正确；前端 `npm run build` 通过、无新依赖。**实测发现 `metadata.writes` 在 langgraph>=1.2 不存在，改用 `snapshot.tasks[*].name` 作节点标签**。
- **待验证（需目标环境）**：Postgres 持久 + 跨重启、多 worker 共享、完整 run 的丰富 state、浏览器交互（时间轴点击 / 刷新重载）、HTTP 层 SSE `thread_id` 回传——当前 dev 机代码在容器外、DB 在容器内、8000 是旧镜像，HTTP 端到端未跑。
- **相关任务**：[`../.trellis/tasks/07-02-graph-debug-time-travel`](../.trellis/tasks/07-02-graph-debug-time-travel/)。

## Graph Debug fork（A2）：改 state + 从 checkpoint 重跑

- **类型**：功能升级（开发者调试工具，A2 fork 范围）。
- **影响范围**：仅 Graph Debug 调试路径；不改 A1 端点 / 响应形状（仅 `/run` SSE 内部解包 bug 修复，事件形状不变）；不动 DB、生产路径。
- **改动明细**：
  - 后端 `POST .../threads/{tid}/checkpoints/{cid}/state`：`aupdate_state`（不传 as_node，config 含 `checkpoint_ns=""`）改 state，返回新 checkpoint_id（同 thread 分支）。
  - 后端 `POST .../checkpoints/{cid}/replay`：`astream(None, config)` 从该 checkpoint 重跑后半段，SSE 复用 `/run` 事件。
  - 抽 `_run_graph_sse` 让 `/run` 与 replay 共用流式逻辑；**修复 A1 的 tuple 解包 bug**（`astream` list-mode 返回 tuple，原按 dict 解 → /run 只发 error）。

- **原因**：让开发者做"换 message/prompt/retrieval 结果，后半段会怎样"的假设实验。fork 结果同 thread 分支（F1）。
- **验证**：进程内（InMemorySaver）全过——`/run` 现发 `node_start/node_end`（tuple 修复确认）、update 生成新 checkpoint、403；前端 `npm run build` 通过、无新依赖。
- **发现（已记 lessons #21）**：① `as_node` 不传即可；② config 必须含 `checkpoint_ns=""`；③ `metadata.parents` 实测空（A3 不能依赖）；④ A1 `/run` SSE tuple bug（本次修复）。
- **待验证（需目标环境）**：HTTP/浏览器 fork 流程、跨重启持久、replay 真实 LLM 重跑。

## 🚨 P0 Hotfix：移除 deepeval `@observe` 装饰器，修复 `/agent/chat` 全租户 500

- **改动对象**：`src/sales_agent/services/chat_pipeline.py`、`task_router.py`、`agent_executor.py`、`retriever.py`（各删 `from deepeval.tracing import observe` import + `@observe(type=...)` 装饰器，共 5 处）。
- **类型**：P0 bug 修复（生产故障）。
- **影响范围**：整个 chat 请求链（`/agent/chat` + 钉钉 stream，都走 `ChatPipeline.execute()`）。移除的是 deepeval 追踪装饰器，不改任何业务逻辑；eval 评分不受影响（eval 直接用 deepeval metrics，不依赖运行时 @observe span）。
- **故障现象**：所有租户 `/agent/chat` 返回 `500 INTERNAL_ERROR "dictionary changed size during iteration"`。实测 prod2(taishan+taishankaifa2)、prod3(songbai)、test(fuduoduo) **全站 500**，生产销售 agent 不回消息。
- **根因**：`ChatPipeline.execute()` 上的 `@observe(type="agent")` 在函数退出时（`Observer.__exit__`）遍历子 span，调用 deepeval `make_json_serializable(child.input)` → `_serialize` 迭代某嵌套对象的**活 `__dict__`**（`vars(obj).items()`），而序列化过程触发该对象惰性字段写回同一 `__dict__` → `RuntimeError: dictionary changed size during iteration`。聊天本身已算完，是装饰器收尾序列化时炸。
- **为何现在才爆**：`@observe` 在 `bb2b1eb feat/fronted-mermaid` 进 main，潜伏至今；prod 未设 `DEEPEVAL_*/Confident` key（日志「Skipping trace posting」），trace 算了又全丢弃——**纯累赘还崩**。
- **改动明细**：移除运行路径全部 5 处 deepeval `@observe`（agent/tool/llm/retriever×2）+ 对应 import；保留 `chat_pipeline.py` 的 `coach_observe`（项目本地实现，非 deepeval）。
- **验证**：`docker cp` 4 文件进 taishan-api + 重启后 `curl /agent/chat` 返回 **200 + 正常回答**（修复前同请求 500）。py_compile 4 文件过；无残留 `from deepeval.tracing import observe` / `@observe`。
- **待办**：构建新镜像并部署到 prod3/test/prod2(taishankaifa2)——这三台仍跑旧镜像、仍在 500。taishan-api 已临时 docker cp 修复（容器层，非镜像，下次重建会丢，故需正式部署）。
- **相关**：lessons #25（deepeval @observe 教训）。
