## Console 前端 dev 连接后端端口修复

### 概述
修复 `console` 下 `npm run dev` 报「当前实例尚未配置 Agent」+
`net::ERR_CONNECTION_REFUSED http://localhost:800X/instance/agent` 的问题。两层根因：

1. **端口错配**：前端 dev 指向宿主机 8001，而本机 8001 被**另一个项目** `ai-coach-fastapi`
   占用（404，非 sales-agent）；sales-agent 后端实际按 `deploy/tenants.json`（`api_port: 8003`）
   跑在 **8003**（`curl :8003/instance/agent` → 200）。
2. **绝对地址对远程浏览器无效**：把 env 改成 `http://localhost:8003` 后，服务器侧 curl 仍 200，
   但浏览器里的 `localhost:8003` 指向**用户自己的机器**（浏览器与服务器不同机）→ 依旧 refused。
   说明 dev 不能用 `http://localhost:<port>` 直连，必须让浏览器走**同源相对路径**，由 vite 在
   **服务器侧**代理到后端。

前端 fetch 失败后 `App.tsx` 的 `InstanceEntry` 一律用「未配置 Agent」兜底，造成误诊
（Agent 实际存在且 active）。

### 改动明细

#### 1. vite dev 代理（核心修复）
- **改动对象**：`console/vite.config.ts`（`server.proxy['/api']`）
- **类型**：配置 / bug 修复
- **影响范围**：本地前端开发
- **改动明细**：代理 target 由 `http://localhost:8000` 改为 `http://localhost:8003`（与本机后端一致），
  保留 `rewrite` 去掉 `/api` 前缀；补注释说明为何用代理（远程浏览器同机直连不可达）。
- **原因**：恢复项目本意的 dev 架构——浏览器同源 `/api/*`，由 vite 转发到后端，无论浏览器是否
  与服务器同机都成立。

#### 2. 前端 dev API 基址改同源前缀
- **改动对象**：`console/.env.development`
- **类型**：配置 / bug 修复
- **影响范围**：本地前端开发
- **改动明细**：`VITE_API_BASE_URL` 由绝对地址 `http://localhost:8001`（后曾试 `:8003`）改为
  同源前缀 `/api`；配合 #1，浏览器请求 `/api/instance/agent` → vite 转发到 `:8003/instance/agent`。
  生产构建仍同源（`.env.production` 留空），行为不变。
- **原因**：消除对 `localhost:<port>` 的依赖，根治远程浏览器 refused。

#### 3. 手写 dev compose（未改，说明）
- **改动对象**：`docker-compose.yml`（`taishan-api`，profile `taishan-split`）保持 `8001:8000`。
- **说明**：本机 8001 实际被 ai-coach 占用，运行中的 sales-agent 后端在 **8003**
  （由 `deploy/tenants.json` 生成的 `docker-compose.generated.yml` 决定）。本次**不改** compose：
  前端通过 vite 代理直连运行中的 8003，不依赖手写 compose 的端口。手写 compose 端口口径
  与生成式部署不一致是既有现象（如在本机用 `--profile taishan-split up` 会与 ai-coach 抢 8001），
  属独立问题，单独跟进。

### 验证
- 启动 `npm run dev` 后，`curl http://localhost:<vite端口>/api/instance/agent`（即浏览器将发出的
  同源请求）→ **200**，返回 taishan Agent（id/tenant_id/status=active）——证明代理对任意能访问
  vite 的浏览器都成立。
- 从转换后的 `src/api/client.ts` 核实 vite 注入 `VITE_API_BASE_URL="/api"`。
- `console` 测试套件 `npm test` → 10/10 通过。

### 影响边界 / 不改动
- **生产配置不动**：`deploy/tenants.prod3.json`（`api_port: 8001`）、
  `scripts/deploy-image-retag.sh`（默认 8001）、prod 部署文档均为**另一台机器**，
  其上 8001 正确，不在本次范围。
- `console/.env.production` 仍为同源（空），生产构建行为不变。

### 未完成 / 后续

  一律显示「未配置 Agent」，易把连通性问题误判为业务缺数据。建议区分「网络错误」与

## deploy-release.sh 交互菜单方向键崩溃修复

### 改动对象
`scripts/deploy-release.sh`（交互式租户选择菜单的按键读取循环，第 169–203 行附近）。

### 类型
bug 修复

### 影响范围
多租户部署脚本的交互式菜单（`secrets/` 下存在 ≥2 个 `.env` 时触发）。

### 改动明细
- **根因**：菜单默认高亮首项 `taishan.env`（`selected=0`）。按 ↓（Down）执行 `((selected++))`，
  这是**后置自增**，表达式取值为旧值 `0`；在 `set -e`（`set -euo pipefail`）下 `(( ))`
  返回 `0` 即退出码 `1`，直接终止整个脚本——表现为「按一下方向键就退出，无任何提示」。
- **修复**：把 `((selected++))` / `((selected--))` 改为算术赋值
  `selected=$((selected + 1))` / `selected=$((selected - 1))`（赋值恒为退出码 0，不受 `set -e` 影响）。
- **加固**：`IFS= read -rsn1 key` 在 stdin 关闭/Ctrl-D 时返回非零，同样会被 `set -e` 终止；
  补 `|| { 恢复光标; 干净退出; }` 兜底，避免 Ctrl-D 崩溃。

### 原因
首次按 ↓（最常见的操作——从首项移到第二个租户或 All）必然触发崩溃，菜单实际不可用。

### 验证
- `bash -n scripts/deploy-release.sh` → 语法 OK。
- 复现 harness：修复前 `↓+Enter` → `rc=1`（崩溃）；修复后 → `rc=0`，正确选中第二项
  `taishankaifa2.env`；`↓↓+Enter` → All；`↓↓↓+Enter` → Quit；`↑` / `q` / ESC 后 EOF 均正常。
- 真实脚本端到端：用 `printf '\x1b[B\x1b[A\nno\n'`（↓ → ↑ → Enter 选 taishan.env，确认步给 `no`）
  驱动，菜单正确移动光标并打印 `Selected: taishan.env`，顺利走到 `Type DEPLOY` 确认门
  （随后因输入非 `DEPLOY` 而按预期 abort）。脚本运行仅写 `mktemp` 临时清单，**未改** `deploy/tenants.json`。

### 附带发现（未在本次处理）
- `secrets/taishankaifa2.env` 已存在，但**尚未**登记进 `deploy/tenants.json`（现仅 `taishan`）。
  因此即便菜单修好，单独选 `taishankaifa2` 会在「清单过滤」步骤报 `No tenants match`。需在
  inventory 补该租户条目后才能部署。
- 本机 8003/8004 端口已被占用，烟雾测试中脚本把临时清单里 taishan 的端口顶到 8005
  （不影响真实 inventory）。

## 第二租户 taishankaifa2 部署级隔离（dedicated 双实例）

### 概述
在 dedicated + 共享 PG 库架构下新增第二租户 `taishankaifa2`（泰山兄弟开发版2），与 `taishan`

### 改动明细

#### 1. compose 新增 taishankaifa2-split 服务
- **改动对象**：`docker-compose.yml`（新增 `taishankaifa2-api/stream/worker`，profile `taishankaifa2-split`）
- **类型**：feat / 配置
- **影响范围**：本地多租户部署
- **改动明细**：仿 `taishan-split`，`env_file=./secrets/taishankaifa2.env`，共享 `postgres` 库，
  `taishankaifa2-api` 映射 `8004:8000`，卷 `./data/taishankaifa2` / `./logs/taishankaifa2`。
- **原因**：为第二租户提供独立 dedicated 实例（部署级隔离）。

#### 2. compose 叠加文件（规避宿主 5432 冲突）
- **改动对象**：`docker-compose.taishankaifa2.yml`（新增）
- **类型**：配置
- **影响范围**：起 taishankaifa2 栈
- **改动明细**：`postgres.ports: []`；用法 `docker compose -f docker-compose.yml -f docker-compose.taishankaifa2.yml --profile taishankaifa2-split up -d`。
- **原因**：主 compose 的 `postgres` 映射宿主 5432，但本机 5432 被 `app-postgres-1` 占用；叠加文件清空
  端口映射，使 `sales-agent-db` 与 `docker-compose.generated.yml` 行为一致（仅容器内可达）。

#### 3. 前端 dev 第二入口
- **改动对象**：`console/vite.config.ts`、`console/package.json`
- **类型**：feat
- **影响范围**：本地前端开发
- **改动明细**：vite proxy `target` 参数化为 `process.env.VITE_PROXY_TARGET || 'http://localhost:8003'`；
  `package.json` 新增 `dev:taishankaifa2`（`VITE_PROXY_TARGET=http://localhost:8004 vite --port 4001 --strictPort`）。
- **原因**：一套前端代码、两个 dev 入口——`dev`(4000→8003 taishan) 与 `dev:taishankaifa2`(4001→8004)。

### 运维流程（新增 dedicated 租户的标准步骤）
1. `secrets/<tid>.env`：`TENANT_ID=<tid>` + 独立 key/钉钉机器人/向量集合/数据目录（chmod 600）。
2. compose 加 `<tid>-split` 服务（或用 `deploy/tenants.json` + 生成器）。
3. 起容器（本机需叠加 `docker-compose.taishankaifa2.yml` 规避 5432 冲突）。
4. `POST /tenants {"tenant_id","name"}` 注册进 tenants 表（朝任一实例，全局端点）。
5. **重启 api 容器**：启动钩子 `ensure_default_agents` 遍历 tenants 表给新租户建默认 Agent
   （`POST /tenants` 本身不建 Agent，必须靠启动钩子）。
6. `curl :<port>/instance/agent` 验证返回该 tenant 的 Agent。

## 前端容器化：每租户独立 nginx SPA 容器

### 概述
将前端从"打进后端镜像、FastAPI 同源 serve"改为**独立 nginx 容器**，CI/CD 部署时每个租户
自动配一个对应的前端容器。前端容器（nginx:alpine）serve SPA + 同源代理 `/api` 到配对的后端
容器。Traefik 用户域名指向前端容器，钉钉回调仍直连后端 API。

- **`console/docker-entrypoint.sh`**：启动脚本（envsubst `$BACKEND_HOST` → nginx conf + exec nginx）
- **`docker-compose.taishankaifa2.yml`**：叠加文件，清空 postgres 宿主端口映射（本机 5432 冲突）

### 修改文件
- **`scripts/render-multitenant-deploy.py`**：
  - `render_tenant_services()`：api role 的 tenant 追加 `{tid}-frontend` 服务（nginx，port=api_port+3000:80，env BACKEND_HOST={tid}-api）
  - `render_traefik_routes()`：用户域名路由 target 从 `api:8000` 改为 `frontend:80`；钉钉路由保持 `api:8000`
- **`scripts/deploy-release.sh`**：新增 Section 8b "Load frontend image"（registry pull / 本地 build / 默认 tag）
- **`scripts/ci-fanout.sh`**：本地/远程执行时传递 `FRONTEND_IMAGE` 给 `deploy-release.sh`
- **`.gitea/workflows/deploy.yml`**：`build-and-push` job 新增前端镜像构建+推送；`deploy-fanout` job 传 `FRONTEND_IMAGE`

### 端口约定
前端端口自动推导：`frontend_port = api_port + 3000`（taishan api=8003 → frontend=11003，taishankaifa2 api=8004 → frontend=11004）。可通过 tenant 配置的 `frontend_port` 字段覆盖。

## 实例配置仪表盘（4000 概览展示 secrets 运行时配置）

### 概述
dedicated mode 运营面板（`/`、`/dashboard`，端口 4000 入口）新增「环境配置」区域，
按分组折叠展示 `secrets/<tenant>.env` 注入的运行时参数。API key / secret / token 类敏感字段
默认掩码（`●●●●●●`），点击 👁 切换截断明文 + 一键复制（5 秒自动恢复掩码）；
AppKey / AppSecret / CorpID 等钉钉集成参数同样展示（AppSecret 仍按敏感规则掩码）。

### 改动明细

#### 1. 后端：TenantRuntime 暴露运行时 env
- **改动对象**：`src/sales_agent/core/tenant_runtime.py`
- **类型**：feat
- **影响范围**：运行时配置读取
- **改动明细**：新增 `TenantRuntime.get_all_env_vars()`，用**白名单前缀**（`DEPLOYMENT_`/`TENANT_`/
  `MODEL_`/`EMBEDDING_`/`VECTOR_`/`DATA_DIR`/`LOG_DIR`/`DINGTALK_`/`NEO4J_`/`ONTOLOGY_`/`COACH_`）
  返回应用相关环境变量。
- **原因**：为配置展示提供数据源。

#### 2. 后端：`GET /instance/config` 端点
- **改动对象**：`src/sales_agent/api/routes/instance.py`（新）、`src/sales_agent/main.py`
- **类型**：feat / API
- **影响范围**：HTTP 接口
- **改动明细**：新增 `/instance/config`，从 `TenantRuntime` 读取全部应用 env，按分组（deployment/
  model/storage/dingtalk/media/coach/neo4j/ontology/other）排序返回。字段名匹配 `*_API_KEY`/
  `*_SECRET`/`*_TOKEN`/`*_AES_KEY`/`*_ENCRYPT_TOKEN` 标记 `sensitive:true` 并附 `masked` 截断值，
  `value` 字段保留完整明文供前端点击揭示后复制。`main.py` 注册 instance router。
- **原因**：前端需要一个端点拿到当前实例运行时配置。

#### 3. 前端：类型 + API + 组件
- **改动对象**：`console/src/api/{types,instance,index}.ts`、`console/src/components/{SensitiveField,ConfigCard}.tsx`
- **类型**：feat / 前端
- **影响范围**：管理控制台
- **改动明细**：
  - `types.ts` 新增 `SensitiveValue`/`ConfigValue`/`InstanceConfigGroup`/`InstanceConfigResponse`；
    `instance.ts` 新增 `getInstanceConfig()`；`index.ts` 导出。
  - `SensitiveField`：掩码圆点 + 👁 揭示 + 📋 复制（写完整 value）+ 5 秒自动恢复。
  - `ConfigCard`：Ant Design `Collapse` 分组折叠，标题含图标 + 中文标签 + 字段数 + 2-3 关键值摘要，
    敏感字段用 `SensitiveField`，非敏感用 `Text`；处理 loading/error/empty 三态。
- **原因**：把运行时配置可视化为可折叠、敏感可控的卡片。

#### 4. 前端：DashboardPage 接入 + 路由
- **改动对象**：`console/src/pages/Dashboard/DashboardPage.tsx`、`console/src/App.tsx`
- **类型**：feat / 前端
- **影响范围**：路由 + 运营面板
- **改动明细**：`DashboardPage` 新增独立 `configQuery`（不纳入 `isLoading` 主门控，配置加载失败不阻塞运营指标），
  底部渲染 `ConfigCard`；`App.tsx` 将 `/` 和 `/dashboard` 从重定向 `InstanceEntry` 改为直接渲染 `DashboardPage`
  （`AgentOverviewPage` 仍保留在 `/agents/:id/overview`）。
- **原因**：让 4000 入口落地为运营指标 + 环境配置的综合面板。

### 安全说明
初版 `get_all_env_vars()` 采用黑名单过滤，会把整个宿主环境泄漏到 `/instance/config`——其中
`ANTHROPIC_AUTH_TOKEN`、IDE/SSH token 等匹配 `*_TOKEN` 后其完整 `value` 会发到前端（一键揭示即可读取），
构成密钥泄漏。**已改为白名单前缀过滤**（仅暴露已知应用配置），Task 9 集成烟雾测试捕获并验证修复：
返回仅含 9 个应用配置变量，`ANTHROPIC_AUTH_TOKEN`/`SSH_CONNECTION` 等宿主密钥不在响应中。

  → 200，分组顺序 `deployment → model → dingtalk`，敏感字段 `sensitive:true`+masked，非敏感为字符串，
  响应体无宿主密钥。
