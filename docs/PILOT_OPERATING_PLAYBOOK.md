# Pilot Operating Playbook — Pilot 运营手册

**版本:** Phase D
**更新日期:** 2026-06-11

## 概述

本手册定义了 Pilot 阶段的日常运营工作流，包括每日质量审查、知识缺口闭环、回归测试、每周报告和 Pilot 退出决策。

---

## 工作流 1：每日质量审查

**频率：** 每个工作日
**角色：** 管理员 / 质量负责人
**页面：** 质量审查 (`/review`)

### 步骤

1. **打开质量审查页面**
   - 点击左侧导航「质量审查」
   - 查看待审查条目数量

2. **扫描新条目**
   - 点击「立即扫描」按钮
   - 系统自动检测：负反馈、高风险、模型错误、检索缺失
   - 记录新增条目数

3. **逐条审查**
   - 按优先级从高到低处理
   - 点击会话 ID 查看完整对话
   - 判断问题根因

4. **分类反馈**
   - 对负反馈条目分配根因分类：
     - `wrong_answer` — 回答错误
     - `missing_knowledge` — 知识缺失
     - `bad_retrieval` — 检索不准
     - `bad_prompt` — Prompt 问题
     - `wrong_task_route` — 任务路由错误
     - `unsafe_answer` — 不安全回答
     - `too_generic` — 过于笼统
     - `too_slow` — 响应太慢
     - `format_problem` — 格式问题

5. **创建知识缺口（如适用）**
   - 如果根因是知识缺失，在「知识缺口」页面创建条目
   - 标题描述缺失的知识领域
   - 关联来源会话

6. **更新状态**
   - 审查完成 → 标记为「已解决」
   - 不需要处理 → 标记为「忽略」

### 完成标准

- 所有 `critical` 优先级条目已审查
- 负反馈已分类
- 知识缺口已创建（如适用）

## 工作流 2：知识缺口闭环

**频率：** 根据需要
**角色：** 管理员 / 知识负责人
**页面：** 知识缺口 (`/gaps`)

### 步骤

1. **查看待处理缺口**
   - 打开「知识缺口」页面
   - 筛选状态为「待补充」或「需要文档」

2. **识别高频缺口**
   - 多个反馈指向同一知识领域
   - 同类检索缺失反复出现

3. **准备文档**
   - 编写或收集相关知识文档（Markdown 格式）
   - 内容应覆盖缺口描述的问题

4. **上传文档**
   - 在「知识库」页面上传
   - 等待导入完成

5. **关联文档**
   - 回到「知识缺口」页面
   - 点击「关联文档」
   - 输入新上传的文档 ID

6. **验证**
   - 使用 Prompt 预览测试相关问题
   - 确认回答引用了新上传的知识
   - 将缺口状态更新为「已验证」

### 状态流转

```
open → document_needed → uploaded → verified
  └── ignored (任何状态可跳到忽略)
```

## 工作流 3：Prompt/RAG 回归检查

**频率：** 每次 Prompt 或知识库变更后
**角色：** 管理员 / 技术负责人
**页面：** Eval 回归 (`/eval`)

### 步骤

1. **记录基线**
   - 变更前运行评估套件
   - 记录运行 ID 作为 `before_run_id`

2. **执行变更**
   - 激活新的 Prompt 版本
   - 或上传新的知识文档

3. **运行评估**
   - 在「Eval 回归」页面运行同一套件
   - 记录运行 ID 作为 `after_run_id`

4. **对比结果**
   - 查看两次运行的通过/失败数
   - 关注回归（之前通过现在失败的用例）

5. **决策**
   - 通过率持平或提高 → 保留变更
   - 出现回归 → 排查原因，可能回滚
   - 回滚方式：将之前版本的 Prompt 重新激活

### 示例

```
Before: 42 passed, 3 failed (93.3%)
After:  40 passed, 5 failed (88.9%)  → 出现2个回归，需要修复
```

## 工作流 4：每周 Pilot 报告

**频率：** 每周一
**角色：** Pilot 负责人
**页面：** Pilot 报告 (`/reports`)

### 步骤

1. **生成报告**
   - 打开「Pilot 报告」页面
   - 选择上周的时间范围
   - 点击生成周报

2. **审阅报告内容**
   - **使用概要**：DAU、会话数、用户活跃度
   - **任务分布**：哪些任务类型使用最多
   - **质量指标**：正反馈比、RAG 使用率
   - **性能指标**：延迟 P50/P95、错误率
   - **反馈根因**：主要问题类别
   - **知识缺口**：待补充和已验证数量
   - **告警**：活跃告警数

3. **准备行动项**
   - 识别需要改进的领域
   - 分配知识缺口补充任务
   - 安排 Prompt 优化工作

4. **分享报告**
   - 下载 Markdown 格式
   - 发送给相关干系人

## 工作流 5：Pilot 退出决策

**频率：** 每 2-4 周评估一次
**角色：** Pilot 负责人 + 业务负责人
**页面：** Pilot 指标 (`/pilot`)

### 决策框架

系统自动计算 Pilot 状态分类：

| 分类 | 含义 | 典型条件 |
|------|------|----------|
| **expand** | 可扩展至正式采购 | 用户活跃、反馈好、无重大问题 |
| **continue_pilot** | 继续 Pilot | 指标尚可，但未达扩展标准 |
| **needs_remediation** | 需要修复问题 | 反馈差、知识缺口多、有严重告警 |
| **stop** | 暂停 Pilot | 激活极低、错误率高 |

### 关键指标阈值

| 指标 | expand | continue | needs_remediation | stop |
|------|--------|----------|-------------------|------|
| 活跃用户 | ≥ 5 | ≥ 3 | 任意 | < 3 |
| 重复使用率 | ≥ 40% | ≥ 20% | 任意 | 任意 |
| 正反馈比 | ≥ 70% | ≥ 50% | < 50% | < 30% |
| 未解决缺口 | ≤ 2 | ≤ 5 | > 5 | 任意 |
| 严重告警 | 0 | ≤ 1 | ≥ 1 | 任意 |
| 错误率 | < 5% | < 10% | 任意 | > 20% |

### 决策流程

1. 打开「Pilot 指标」页面查看自动分类
2. 验证系统分类是否合理
3. 与业务负责人讨论
4. 根据分类采取行动：
   - `expand` → 准备合同和扩大部署
   - `continue_pilot` → 继续运营，下周再评估
   - `needs_remediation` → 优先修复阻塞问题
   - `stop` → 与客户沟通暂停，评估产品匹配度
