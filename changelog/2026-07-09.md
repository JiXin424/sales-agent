## Eval 租户绑定修复：graph-multiturn / dingtalk-staging 独立 CLI 跑通 chat-path（Spec 4 §3.2,§3.4）

- **改动对象**：`eval/memory_eval/runner.py`（`_bind_test_database` / `_restore_database`）、`tests/integration/test_memory_eval_graph_multiturn.py`、`tests/integration/test_memory_eval_dingtalk_staging.py`。
- **类型**：bug 修复（评估 harness 独立运行缺陷）。
- **影响范围**：仅评估 harness 离线入口；**不改动在线 Graph 请求路径**，生产流量零影响。
- **改动明细**：
  - **根因**：`graph-multiturn` / `dingtalk-staging` 独立 CLI 跑测试库时，chat-path 场景路由到 Graph 的 `resolve_tenant` 节点 → `TenantResolver.resolve` → `get_tenant_runtime().check_tenant_match(tenant_id)`；dedicated 模式下 runtime 租户取自 `TENANT_ID` 环境变量，与 `args.tenant_id`（`eval-tenant`）不等，抛 `TenantMismatchError`，场景失败。既有集成测试只用 explicit-remember 场景（走 `memory_command`，绕过 `resolve_tenant`），未覆盖此路径。
  - **修复**：`_bind_test_database(test_db, tenant_id)` 在绑定测试库时同步把 `os.environ["TENANT_ID"]` 设为 eval 租户并置空 `tenant_runtime._runtime` 单例（`load_dotenv` 不覆盖已存在 env，故此处设置胜出）；`_restore_database` 恢复原 `TENANT_ID` 并再次置空单例，避免泄漏。`run_graph_multiturn` / `run_dingtalk_staging` 两处调用点传入 `args.tenant_id`。`_seed_eval_fixtures` 已种子该租户行，故 `TenantResolver.resolve` 可命中。
  - **测试加固**：两个集成测试的内联数据集各新增一个 chat-path 场景（如「帮我看看福多多产品怎么讲」，`turn_relation:"new"`），端到端触发 `tenant_resolve`；`test_memory_eval_graph_multiturn.py` 新增 `test_graph_multiturn_chat_path_runs_past_tenant_resolve` 直接断言 chat turn `response_kind != "duplicate"` 且无 error（因聚合报告不序列化逐轮 error，ScenarioRunner 会吞掉异常）。dingtalk-staging 的 turn 循环无 try/except，故 tenant 崩溃会直接传播使测试失败——加 `total_scenarios == 2` 断言即足够守卫；chat 场景 event_id 用 uuid 后缀避免 `memory_outbox` 唯一约束在持久测试库上重跑碰撞。
- **原因**：补齐 Spec 4 头条交付物「一条命令跑 ≥40 场景」——修复前 chat-path 全部崩在 tenant_resolve，报告 hollow。
- **验证**：独立 CLI `graph-multiturn` 跑完 42 场景无 `TenantMismatchError`，非 hollow（`topic_leakage_rate 13/52`、`explicit_operation_accuracy 10/26`）；门禁 Phase B 仅因质量门禁失败（零 tenant 相关失败）；`pytest tests/unit/eval tests/integration/test_memory_eval_graph_multiturn.py tests/integration/test_memory_eval_dingtalk_staging.py -q` → 82 passed；模拟未绑定租户确认 `run.error` 为 TenantMismatchError（回归守卫有效）。

## 钉钉语音/图片支持流式互动卡片

- **改动对象**：`src/sales_agent/integrations/dingtalk/stream_client.py`、`src/sales_agent/integrations/dingtalk/graph_stream.py`；新增测试 `tests/unit/dingtalk/test_stream_client.py`。
- **类型**：bug 修复 + 功能补全（让语音/图片与文字享受同一流式卡片路径）。
- **影响范围**：仅钉钉 **Stream** 模式（生产主入口），且仅在 `dingtalk.streaming_enabled=true`（默认关）+ `dingtalk.media_enabled=true`（默认开）时生效。HTTP 回调模式、关闭流式的实例行为零变化。无 DB 变更、无 Alembic migration、无配置 schema 变化。
- **改动明细**：
  - **根因**：`stream_client.py` 的流式门禁硬编码 `message_type == "text"`，语音(`voice`/`audio`)/图片(`picture`/`image`/`richText`)被直接挡进非流式 `handle_dingtalk_event`；且流式分支 `_handle_streaming` 虽接收 `media_download_codes`/`raw_event` 参数却从未使用，也没调 `DingTalkMediaAdapter`——媒体转不出文字，是没接完的半成品。
  - **改动①门禁**：抽出模块级纯函数 `_should_stream(config, message_type, text_content)`——文字（非快速命令）或受支持媒体（`media_enabled` 开）才进流式，其余不变。`process()` 内 `use_streaming` 改调它。
  - **改动②透传类型**：`_handle_streaming` 签名 + 调用处补 `message_type` 参数（此前未传）。
  - **改动③接媒体适配**：新增模块级 `_resolve_streaming_message(...)`——文字直接透传不开卡；媒体先开「正在识别…」过渡卡片（阻塞 ASR/视觉期间给用户反馈），再 `DingTalkMediaAdapter.to_agent_text` 转写，返回 `(text, card_id)`；转写失败在**同一张卡** `streaming_finalize` 友好提示（`reply_fn` 仅作卡片 finalize 也失败时的兜底）。`_handle_streaming` 调用它，`message_text is None` 时直接 return 不进 Graph。
  - **改动④复用卡片**：`handle_dingtalk_stream_via_graph` 加可选 `card_id: str | None = None`——传入则复用过渡卡、不新建「分析中…」卡，实现「正在识别 → 分析中 → 回答」单卡 UX；不传则维持原行为（文字路径零回归）。
  - **改动⑤修兜底**：`_handle_streaming` 在 `CardSender` 不可用降级时，原先硬写 `message_type="text"` 且丢弃 `media_download_codes`/`raw_event`，导致媒体消息无法转写；改为透传真实 `message_type` 与媒体字段，落到 `handle_dingtalk_event` 的媒体分支兜底。
  - **未夹带**：token 级流式隐患（生成节点 `generate_node` 经 `execute_agent` → `chat_model.generate()` 为非流式调用，真正的 `stream_generate` 全仓无调用方；`graph_stream.py` 中 `GRAPH_CHUNK`/`GRAPH_DONE`/`CARD_FINALIZE` 满屏 `logger.warning` 调试日志印证团队已在排查）属独立问题，本次不动，留作后续任务。
- **原因**：用户反馈「钉钉发语音/图片不能流式，只有文字能流式」。这是显式架构门禁 + 流式路径未接媒体的组合问题，非偶发 bug。目标是让媒体走与文字完全相同的流式路径（parity），而不是另起一套。

  - 回归：`tests/unit/dingtalk/` 111 项全过；引用改动函数的相关测试 41 项全过。

## 修复钉钉 Stream 连接不稳定（websockets ping 与钉钉网关冲突）

- **改动对象**：`src/sales_agent/integrations/dingtalk/stream_client.py`（import dingtalk_stream 后加 websockets ping 兼容补丁）。
- **类型**：bug 修复（连接稳定性）。
- **影响范围**：所有租户的 stream 容器（taishan/fuduoduo/songbai/taishanyanshi 等）；api/worker 不连 wss 不受影响。
- **改动明细**：monkey-patch `websockets.connect.__init__`，当调用方未显式传 `ping_interval`/`ping_timeout` 时强制设为 `None`（幂等，标记 `_sales_agent_ping_patched`）。
- **根因**：test 镜像 `websockets==15.0.1` 默认 `ping_interval=20` + `ping_timeout=20`，钉钉 wss 网关对高频 ping 响应慢，20s 内无 pong 即 `ConnectionClosedError`，导致 dingtalk_stream SDK `[start] network exception` 反复重连。连接抖动期间钉钉推的消息送不进来（文字/图片/知识库查询全失效，表现为"收不到消息""图片处理不了""知识库无来源"）。SDK 调 `websockets.connect(uri)` 未覆盖 ping 参数，且自带 60s keepalive。prod2 是 websockets 16.0（默认行为不同）所以 taishan stream 稳定。
- **验证**：禁用自带 ping 后容器内独立监听 45s+ 稳定不断（修复前每次启动必 network exception）；`tests/unit/dingtalk/test_stream_client.py` 12 项全过；patch 逻辑单测确认 `ping_interval=None`。
- **部署**：commit `ad5859b` push origin main -> CI fan-out 部署 prod2/test/prod3。
