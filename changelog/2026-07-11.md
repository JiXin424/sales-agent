# 2026-07-11 升级日志

## 销售动作提醒抽取链修复 + 全量启用（Sales Action Reminders）

- **改动对象**：`src/sales_agent/services/sales_actions/parser.py`、`config/llm_config.yaml`、`src/sales_agent/core/config.py`；5 租户 `.env`（部署侧）。
- **类型**：bug 修复（断链）+ 配置能力（env 开关）+ 灰度上线。
- **影响范围**：钉钉 Stream 销售动作提醒功能；CI/CD 连接的 5 个租户（prod2: taishan/taishankaifa2；prod3: taishanyanshi/songbai；test: fuduoduo）。门控默认关，仅显式设 `SALES_ACTIONS_ENABLED=true` 的租户受影响。

### 背景 / 原因

用户对 taishan 钉钉机器人说「5分钟后提醒我上厕所」，机器人回复「收到，5分钟后提醒你上厕所…」，但到点无任何主动提醒。

诊断（非猜测，日志+代码核实）：**这不是「提醒功能的 bug」，而是功能从未真正接通 + 默认关闭，模型在幻觉一个它不具备的能力**。

1. **抽取链断裂**：`parser.py:52` 调用 `get_prompt("task","sales_action_extractor")`，但 (a) 该 prompt 在 `config/llm_config.yaml` 缺失——是 commit 1555b5e「YAML 参数与 prompt 合并为 llm_config.yaml」重构时遗漏；(b) `parser.py` 顶部未 import `get_prompt`；(c) 该调用在 `parse_sales_action_request` 的 `try` 块外，异常会直接上浮而非按 docstring 承诺降级。→ 一旦启用，用户会看到「销售动作处理失败」。
2. **默认关闭**：`SalesActionsConfig.enabled=False`，无任何 env/YAML 打开。所有「提醒我」消息在 `normalize_turn_node` 不命中 `sales_action` 分支，落 `flow_action="chat"` 普通聊天 → LLM 凭训练经验幻觉答应，后端零动作。生产出站文案「收到，5分钟后提醒你上厕所…」既非建提醒模板 `已创建提醒：…` 也非失败模板，佐证走的是 chat 路径。
3. 基础设施（DB 表 0016、repository、scheduler、钉钉 card_sender 主动推送、graph 节点与边）已存在且已接线，0016 迁移已落库。缺的只是接通触发点 + 打开开关。

### 改动明细

- `parser.py`：新增 `from sales_agent.llm.prompt_loader import get_prompt`；`parse_sales_action_request` 把 `messages = _build_messages(...)` 移入 `try` 块，使 prompt/loader 异常也降级为 `intent="none"`（回退普通聊天）而非抛「处理失败」，兑现文件 docstring 的「任何失败都不抛异常」契约。
- `config/llm_config.yaml`：`prompts.task` 下新增 `sales_action_extractor`（判断 create/suggest/complete/cancel/list 意图 → 结构化 JSON，含 5min/相对时间/模糊时间 few-shot）。字段与主干 `SalesActionExtraction` 契约吻合。因 `parser.py` 直接用 `.template` 不经 `.format()`，JSON 示例花括号保持单括号（不转义）。
- `core/config.py` `from_yaml()`：新增 `SALES_ACTIONS_ENABLED` 环境变量覆盖（仿 `GUIDED_FLOWS_ENABLED` 模式），支持按租户 `.env` 精确开关，代码默认仍 `False`。

### 金丝雀发现并修复的时区 bug（parser now 时区）

taishan 灰度真实验证时，「1分钟后提醒我测试」全部被判 `past_time`（"那个时间已经过去了…"），0 卡创建。根因：`nodes.py:929` 传给抽取器的 `now = datetime.now(timezone.utc)` 是 **UTC**（容器默认时钟，`TZ` 空），但 `_build_messages` 的 prompt 又声明「时区：Asia/Shanghai」。LLM 据此把相对时间按 +08:00 输出同一钟点数，换算回 UTC 反而**早 8 小时**，被 `validate_action_extraction` 判过去时间拒绝建提醒。修复：`parser._build_messages` 把 `now` 统一 `astimezone` 到声明时区再写进 prompt（naive now 先按 UTC 补挂；未知时区名容错退回），使 prompt 自洽。新增 `tests/unit/sales_actions/test_parser_timezone.py` 4 条回归测试守护。

### 金丝雀发现并修复的跨轮合并 bug（missing_fields 复活 + 完整请求旁路）

tz 修复后重测，抽取本身已正确（真实 deepseek 复现：「1分钟后提醒我测试」→ title='测试'、未来时间、missing=[]），但用户仍收到「要我提醒你做什么？请说一下具体的事。」。根因在跨轮澄清合并 `SalesActionService._merge_partial`：`missing_fields=[...] or old.missing_fields` 在本轮已补全（`missing=[]`）时，`or` 回退把**旧的 missing 复活**。于是一旦某轮把 `["title"]` 塞进 checkpoint 的 `pending_partial`，之后每轮合并即使 title 已给出，`validate` 里 `"title" in missing_fields` 仍成立 → 永久 `missing_title`（完美解释「他总是解析不出来」）。两处修复：
- `_merge_partial`：`missing_fields` 改为「两轮报缺并集 − 合并后已填上的字段」，与合并后实际值一致，不再复活。
- `_extract_with_merge`：本轮若已是完整显式建提醒（title+时间+explicit、无需澄清）则**旁路合并**、直接采用并丢弃旧 partial——避免陈旧 partial 的空标题/过期时间污染一个本来完整的新请求（该场景也会让 `scheduled_at=old or new` 保留过期时间再变 `past_time`）。澄清答复（如「明天下午3点」缺标题）不触发旁路，仍正常补全。
- 新增 `tests/unit/sales_actions/test_merge_partial.py`（复活/保留/补全/旁路 4 场景）。

## 提醒卡片「一直加载中」修复（scheduler 缺流式结束帧）

- **改动对象**：`src/sales_agent/services/sales_actions/scheduler.py`（`_deliver_single` / `_deliver_digest`）。
- **类型**：bug 修复（钉钉流式卡片渲染）。
- **影响范围**：所有启用租户的提醒/digest 卡片投递（5 租户）。

### 现象 / 根因

提醒能建、到点也投递（DB `deliveries.status=success`、拿到 `card_instance_id`），但钉钉卡片**一直显示"加载中"不出内容**。根因：提醒卡片模板是钉钉 **AI 流式卡片**（`createAndDeliver` 带 `callbackType=STREAM`），这类卡片创建后处于"生成中"态，必须再发一个 **`streaming_finalize` 结束帧**（`isFinalize=true`）才会关闭"生成中"指示器并定格内容。普通聊天回复正常，是因为走了完整 `send_markdown_card → streaming_update → streaming_finalize`（`graph_stream.py:171/67/250`）；而 scheduler 的 `_deliver_single`/`_deliver_digest` 只 `send_markdown_card`、从不 finalize（`scheduler.py:273/326`），故卡片永远转圈。DB 记成功是因为 `createAndDeliver` 本身成功返回。

### 改动明细

- `_deliver_single` / `_deliver_digest`：在 `send_markdown_card` 之后、`record_delivery_success` 之前补 `await sender.streaming_finalize(out_track_id, md)`（放在 try 内，finalize 失败则记 failure 走重试）。模式同 `stream_client.py:100→121` 的"创建后立即 finalize"兜底路径。
- 测试：`tests/integration/test_sales_action_scheduler.py` 的 `FakeSender` 增 `streaming_finalize` 捕获 + `test_due_reminder_delivered_once` 断言 `sender.finalizes == [(out, md)]`（回归守护）。

### 验证

- 单测 68 passed；集成测试 8 passed（专用 `sales_agent_test` 库，未碰生产）。
- 生产入口（钉钉 Stream）真机复验：见任务 `.trellis/tasks`。

## 提醒跨租户「串台」修复（scheduler 按 TENANT_ID 收敛）

- **改动对象**：`services/sales_actions/repository.py`（`claim_due_reminders` / `list_active_action_scopes` / `find_overdue_pursuit_actions`）、`services/sales_actions/scheduler.py`（`sales_action_scheduler_loop` / `run_sales_action_scheduler_once` / `_create_due_digests`）、`roles/worker_runner.py`。
- **类型**：bug 修复（多租户隔离）。
- **影响范围**：共享 postgres 库的租户对——prod2(taishan/taishankaifa2)、prod3(taishanyanshi/songbai)。fuduoduo(test) 独立库不受影响。

### 现象 / 根因

用户在 taishan 建的提醒，到点却由 **taishankaifa2 的机器人**发出来（串台）。根因：taishan 与 taishankaifa2 **共享同一 postgres 库**但**各跑一个 worker**，都扫同一张 `sales_action_reminders` 表；`claim_due_reminders` **完全不按 tenant_id 过滤**（docstring 原文"跨租户全局认领"——单一全局调度器的旧假设）。两个 worker 用 `FOR UPDATE SKIP LOCKED` 抢同一批到期提醒，谁先抢到谁投递，且各自用**自己租户的钉钉凭证**（`DingTalkCardSender(settings.dingtalk)`）→ B 租户 worker 抢走 A 租户提醒并用 B 的机器人发出。这是**启用「多租户共享库」功能才暴露的预存隔离缺陷**。

### 改动明细

- 每容器已有 `TENANT_ID` env（`get_tenant_runtime().tenant_id` 读取）。将其从 `worker_runner` 串到 `sales_action_scheduler_loop → run_sales_action_scheduler_once →` 三个 pass。
- `claim_due_reminders` / `list_active_action_scopes` / `find_overdue_pursuit_actions` 新增 `tenant_id: str | None`：**条件式过滤**——给定则 `WHERE tenant_id=?` 只处理本租户；为空则全局（兼容假想的单一全局调度器）。当前所有部署每容器都设 TENANT_ID，故都收敛到本租户。
- 认领(claim)、digest 创建(list_active_action_scopes)、pursuit overdue 三个 pass 一致收敛。
- 测试：`test_sales_action_scheduler.py` 新增 `test_tenant_scoped_run_claims_only_own_tenant`（共库两租户，`tenant_id='t1'` 只投递 t1、t2 状态仍 scheduled）。

### 验证

- 单测 68 passed；集成测试 9 passed（含新隔离测试，专用 `sales_agent_test` 库）。
- 生产复验：prod2 taishan/taishankaifa2 各建提醒，确认各自机器人投递、不再串台——见任务 `.trellis/tasks`。
