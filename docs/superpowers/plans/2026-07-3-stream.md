# Stream-Only Chat and Trusted Optimization Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Replace every chat execution path with one release-aware LangGraph runner, then make DeepEval and the knowledge-QA optimizer evaluate and safely publish real retrieval/prompt candidates through that same graph.

**Architecture:** `ChatGraphRunner` is the sole application boundary around the compiled chat graph. DingTalk Stream calls `astream`, DeepEval calls `ainvoke`, and both resolve either the active immutable release or an explicitly supplied candidate release. Retrieval produces evidence only, common generation produces the answer, and the optimizer compares paired baseline/candidate runs before atomically activating a release.

**Tech Stack:** Python 3.12, LangGraph, FastAPI, SQLAlchemy async, Alembic, PostgreSQL/pgvector, Neo4j, DeepEval, pytest/pytest-asyncio, Docker Compose.

---

## File Responsibility Map

- `src/sales_agent/graph/runtime.py`: typed runtime/release configuration passed to graph nodes.
- `src/sales_agent/graph/runner.py`: sole graph construction and invocation boundary.
- `src/sales_agent/graph/chat_graph.py`: graph topology only.
- `src/sales_agent/graph/nodes/retrieval.py`: evidence retrieval only; no answer generation.
- `src/sales_agent/graph/nodes/generation.py`: common answer generation for RAG and Ontology.
- `src/sales_agent/services/release_service.py`: active/candidate manifest resolution, activation, rollback.
- `src/sales_agent/models/knowledge_version.py`: versioned retrieval profile schema.
- `eval/deepeval_test_cases.py`: graph result to DeepEval test-case adapter.
- `eval/optimizer/metrics.py`: quality score, binary pass rate, paired comparison.
- `eval/optimizer/candidates.py`: constrained retrieval and knowledge-QA prompt candidates.
- `eval/optimizer/publisher.py`: release gate, activation, smoke, rollback.
- `eval/optimizer/graph.py`: persisted optimizer orchestration.

- [ ] **Step 3: Implement typed runtime and runner resolution**

```python
@dataclass(frozen=True)
class GraphRuntimeContext:
    db: AsyncSession
    chat_model: ChatModel
    embedding_model: EmbeddingModel
    tenant_id: str
    agent_id: str
    release: ReleaseManifest
    retrieval_profile: RetrievalProfile
    task_prompts: Mapping[str, str]
    system_prompt: str
    execution_mode: Literal["production", "evaluation", "smoke"]

class ChatGraphRunner:
    async def resolve_runtime(self, *, tenant_id, agent_id, release_id=None, execution_mode="production"):
        release = await self._release_service.resolve_for_run(
            tenant_id=tenant_id, agent_id=agent_id, release_id=release_id,
        )
        ...
```

`resolve_for_run` must tenant/agent-scope an explicit release and otherwise follow `AgentRuntimeBinding.active_release_id`.

- [ ] **Step 4: Add `ainvoke` and `astream` wrappers**

Both wrappers build one compiled graph, pass the same `GraphRuntimeContext`, and require `thread_id`. Production/evaluation missing models raise `GraphRuntimeConfigurationError`; they never return the unit-test fallback answer.
