# 2026-07-10 升级日志

## DeepEval HTML 报告渲染 markdown（修复 agent 回答显示成源码）

- **改动对象**：`eval/deepeval_html_report.py`、新增 `eval/vendor/marked.min.js`（marked v12.0.2，MIT）。
- **类型**：bug 修复（评估工具展示层）。
- **影响范围**：仅 `eval/deepeval_run.sh` 生成的 HTML 报告（`eval/results/deepeval/*.html`）。**不改动在线 Graph 请求路径**、不影响钉钉/HTTP 生产链路、不影响 MD/CSV 报告、无 DB 变更、无配置变更。报告保持自包含（浏览器离线打开即可渲染）。
- **改动明细**：
  - **根因**：`deepeval_html_report.py` 的 HTML 报告是客户端 JS 渲染（数据 `json.dumps` 内嵌、`innerHTML` 拼装），「完整回答」与「参考答案」字段只做了 `.replace(/</g,'&lt;').replace(/>/g,'&gt;')` 尖括号转义，**没有 markdown→html 转换**，导致 `##`/`-`/`**`/`|...|` 全部原样显示成源码。整个报告所有文本字段都如此，回答字段因内容长、markdown 密集而最显眼。
  - **修复①内嵌 marked**：新增 `eval/vendor/marked.min.js`（marked v12.0.2 minified，35KB，UMD）；`_load_marked_js()` 在报告生成时读取并内嵌进 `<script>` 块——源码文件保持干净，报告仍自包含可离线打开。文件缺失时回退纯文本转义，不阻断报告生成。
  - **修复②mdToHtml 渲染**：新增 JS `mdToHtml(text, maxLen)`——**先转义 `&` 和 `<` 防 XSS（刻意保留 `>` 以识别 GFM blockquote），再 `marked.parse(s, {gfm:true, breaks:true})`**。「完整回答」/「参考答案」两字段改调它（含 2000 字截断保持原行为）。配套新增 `.v.md` CSS：标题/列表/表格/代码块/blockquote/链接样式。
  - **不渲染「指标理由」**：`metric_reasons` 是 LLM-as-judge 的自然语言、非 markdown，保留原 `white-space:pre-wrap` 纯文本展示。
- **原因**：用户报告 HTML 报告里 agent 回答的 markdown 不渲染。修复后表格/标题/列表/代码块正常展示，且防住 `<script>` 等 inline HTML 注入。
- **验证**：① `python -m py_compile eval/deepeval_html_report.py` 通过；② 构造含标题/加粗/列表/表格/代码块/blockquote + `<script>alert(1)</script>` 的测试数据生成 HTML，静态确认 marked 库内嵌 + `mdToHtml` 定义 + 两字段调用 mdToHtml；③ **node 实跑 marked + mdToHtml 等价逻辑**：`##`→`<h2>`、`**x**`→`<strong>`、列表→`<ul><li>`、**表格→`<table><thead><th>+<tbody><td>`**、代码块→`<pre><code class="language-python">`、blockquote→`<blockquote>` 全部命中；XSS payload 输出 `<p>&lt;script&gt;alert(1)&lt;/script&gt;</p>`（无可执行 `<script>` 标签）。markdown 渲染仅在浏览器端 JS 执行，Python 端零新增依赖（venv 里的 `markdown-it-py` 是 `rich` 传递依赖，本次未引入）。

## Scenario Coach 钉钉渲染缺分组标题「二、再应对」（修复 loader 丢弃空正文父标题）

- **改动对象**：`src/sales_agent/scenarios/loader.py`、`src/sales_agent/services/response_formatter.py`、`tests/unit/test_scenario_loader.py`、`tests/unit/test_response_formatter.py`。
- **类型**：bug 修复（scenario_coach 渲染的数据层 + 输出层）。
- **影响范围**：仅 scenario_coach 命中时的预设答案渲染（钉钉 stream 卡片 + HTTP `format_text_output`）。不影响路由、RAG、生成、DB。所有「需判断型」场景（Q01/Q02/Q03/Q04…，6 个场景里的大部分）此前都缺「二、再应对」这一行分组标题——正文内容本身不丢，只是缺这行统领标题。
- **改动明细**：
  - **根因**：手册里 `**二、再应对：判断完，对应着打**` 这类分组父标题下面直接接 `####` 子标题、自身没有独立正文。`loader.py` 的 `_split_sections` 把它解析成一个 `content=""` 的 section，随后被末尾的 `[s for s in sections if s.content]` 一刀切丢弃，分组标题永不进入 `answer_dict`，钉钉怎么渲染都出不来这行。`format_text_output` 的 `if title and content` 是第二层同样的空正文过滤。
  - **修复①loader**：`_split_sections` 末尾改成只按 title 过滤（`if s.title`），保留分组父标题（content 可空）。
  - **修复②formatter**：`format_text_output` 对空 content 的 section，作为纯标题行输出（`lines.append(title)`），不再因 content 为空而跳过。
  - **测试**：新增 `test_group_header_without_body_preserved`（loader）、`test_group_header_without_content`（formatter）两个先红后绿回归；放宽 `test_question_fields_populated` 的「每个 section 必有 content」契约（分组标题可无正文）。
- **原因**：用户测试 scenario_coach 命中后，钉钉端渲染缺「二、再应对：判断完，对应着打」这行分组标题（其余内容完整）。一度怀疑流式截断——实为 loader 数据层 bug，命中走的是非流式 `format_text_output`。
- **验证**：① scenario 全套单测 43 passed（`test_scenario_loader/matcher/config/citation` + `graph/test_scenario_coach_node/coach_graph` + `test_response_formatter`）；② 端到端 `parse_scenario_md` + `format_text_output` 跑 Q04，输出含「二、再应对：判断完，对应着打」且四个 `####` 子节正文齐全。
## 联网搜索：部分命中实体缺口补全

- **改动对象**：ontology 检索路径（`graph/chat/nodes/retrieval.py`）、`graph/retrieval/web_fallback.py`、`graph/retrieval/gap_fill.py`（新增）、`prompts/web_analysis_prompt.py`、`core/config.py`。
- **类型**：功能增强（检索层）。
- **影响范围**：hybrid 部署（`KNOWLEDGE_ENGINE=ontology_neo4j` + `HYBRID_RETRIEVAL=true`，生产默认）下，当 query 提到的品牌/产品实体只有部分在知识图谱命中时（如「全品C 和 X品牌 区别」，全品C 在 KB、X品牌 不在）。无 DB 变更、无配置破坏性变更（新字段 `max_gap_entities` 默认 2）。旧 `_retrieve_via_rag` 路径与 `policy=web` 整句域外主路径不动。
- **改动明细**：
  - 新增 `graph/retrieval/gap_fill.py`：纯函数 `is_entity_like`（剔除「区别/对比/怎么样」等非实体词与过短词）/ `is_covered`（双向大小写不敏感子串匹配，对齐 Cypher CONTAINS）/ `compute_missing`（search_terms 减命中实体名，保序去重、`max_n` 封顶）。
  - `_retrieve_via_ontology` 在 `compact_evidence` 之后做缺口检测：对未命中实体定向调 `web_fallback_and_analyze`（查询词 `"{实体} 产品 功能 介绍"`，`context_message` 保留原句意图），web 结果的 context 块与 sources **追加合并**到 KB 结果（取代原「全空才触发 + early-return 整体替换」）。原「全空」case 自然退化为「所有实体 missing -> web 填全部」。
  - `web_fallback_and_analyze` 新增可选 `search_query`（传给 Bocha，缺省退回 message）/ `context_message`（喂分析 LLM 的原句问题，缺省退回 message）；旧 3 个调用点零改动。`web_analysis_prompt` 加 `{user_question}` 段让分析看到用户原始意图。
  - `WebSearchConfig` 新增 `max_gap_entities: int = 2`，env `BOCHA_MAX_GAP_ENTITIES` 可覆盖。
  - web 失败/无结果/未配 key 时静默回退纯 KB（`web_fallback_and_analyze` 返回 None），不阻断主流程。
- **原因**：用户问「全品C 和某品牌 区别」时，全品C 在 KB、某品牌 不在，原管线因 `sources>0`（全品C 在）直接放行、只拿全品C 素材硬答；现对缺失实体走 web 补资料，引用区按来源标「知识库 / 网络搜索」。引用区分来源复用既有 `source_type`（ontology/web），生成侧零改动。
- **验证**：`gap_fill` 7 单测 + `web_search_config` 2 单测 + `web_fallback` 5 单测 + `retrieval_web_fallback` 6 单测（含部分命中合并/全命中不触发/封顶）全过；`test_retrieval_node` 等回归 25 项全过。生产 stream 端到端验证待部署。
