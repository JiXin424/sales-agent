# 2026-06-26 升级日志

## 知识库（ontology_neo4j）支持上传 .doc 与 .xlsx

### 概述
生产知识库走 Neo4j 本体引擎（`POST /agents/{agent_id}/ontology/ingest`），原白名单仅
`.md/.txt/.docx/.pdf/.pptx`。本次新增 **旧版 `.doc`**（OLE2 二进制，`python-docx` 不支持）与
**`.xlsx`** 两种格式的上传解析，解析为纯文本后进入现有 LLM 实体/事实抽取 → Neo4j 入库链路。

- `.doc`：用 **LibreOffice 无头转换**（`soffice --headless --convert-to docx`）转成 `.docx`，
  再复用 python-docx 段落抽取。选 LibreOffice 而非 antiword/catdoc 的原因：中文 `.doc` 用 antiword
  易乱码、会破坏下游 LLM 抽取质量，LibreOffice 中文支持最好。代价：镜像 `+约 150~200MB`
  （`libreoffice-writer`），单文件转换冷启动慢数秒（300s job 超时兜底）。
- `.xlsx`：用 **openpyxl**（`data_only=True` 取公式缓存值、`read_only=True` 省内存），按 sheet
  遍历行，每 sheet 以 `## <sheet名>` 作小标题、行内单元格 tab 连接，保留表格结构供 LLM 理解。

---

### 改动明细

#### 1. 依赖
- **改动对象**：`pyproject.toml`、`Dockerfile`
- **类型**：依赖新增
- **影响范围**：后端镜像构建
- **改动明细**：
  - `pyproject.toml` dependencies 新增 `openpyxl>=3.1.0`（`.xlsx` 读取；项目原只有 `xlsxwriter` 只能写）。
  - `Dockerfile` apt-get install 列表新增 `libreoffice-writer`（含 `soffice` 命令 + Writer 的 `.doc/.docx` 过滤器，足够无头转换）。
- **原因**：旧版 `.doc` 无可靠纯 Python 解析器；openpyxl 是 `.xlsx` 读取的事实标准。

#### 2. 上传白名单
- **改动对象**：`src/sales_agent/api/routes/ontology.py`
- **类型**：配置
- **影响范围**：`/agents/{id}/ontology/ingest` 上传校验
- **改动明细**：`ALLOWED_EXTENSIONS` 由 `{.md,.txt,.docx,.pdf,.pptx}` 扩为 `{.md,.txt,.docx,.doc,.pdf,.pptx,.xlsx}`（校验逻辑集合驱动，无需改）。
- **原因**：原白名单直接 400 拒绝 `.doc/.xlsx`。

#### 3. 文件解析
- **改动对象**：`src/sales_agent/ontology/ingestion_service.py`
- **类型**：功能增强（+ 重构）
- **影响范围**：`_read_content()` 文本抽取
- **改动明细**：
  - 抽出 `_docx_text(path)` helper，`.docx` 分支复用（行为不变，纯重构）。
  - 新增 `_doc_to_text(src)`：每次调用独立 `TemporaryDirectory` + `UserInstallation` profile（避免并发入库的 profile 锁竞争），`subprocess` list 形参调 soffice（文件名含中文/空格也安全），转 `.docx` 后 `_docx_text` 读文本；soffice 未装时抛 `"未安装 LibreOffice"`。
  - 新增 `.xlsx` 分支：openpyxl 按 sheet 遍历，`## <title>` 小标题 + 行单元格 tab 连接。
  - `_read_content` 新增 `.doc` 分发分支；`.docx`/`.doc`/`.xlsx` 失败统一包装 `RuntimeError("xxx 解析失败：<文件名>")`，与现有 pdf/pptx 风格一致。
- **原因**：让两种新格式产出可供 LLM 抽取的纯文本；`_read_content` 仍同步内联调用（与现有 pdf/pptx 一致，300s 超时兜底）。

#### 4. 前端
- **改动对象**：`console/src/pages/Agents/AgentKnowledgePage.tsx`
- **类型**：UI
- **影响范围**：Agent 知识页上传组件
- **改动明细**：`<Dragger accept>` 加 `.doc,.xlsx`；提示文案更新为「支持 .md / .txt / .doc / .docx / .pdf / .pptx / .xlsx」。
- **原因**：前端文件选择器同步放开新格式。

#### 5. 测试
- **改动对象**：`tests/unit/ontology/test_read_content.py`
- **类型**：测试
- **影响范围**：`_read_content` 单测
- **改动明细**：新增 `.doc`（mock `_doc_to_text`）、`.doc` 失败、`_doc_to_text` 缺 soffice 守卫、`.xlsx`（mock openpyxl，校验 sheet 标题 + tab 连接 + 全空行跳过）、`.xlsx` 失败 共 5 个用例；现有 5 个用例回归。
- **原因**：CI 无需装 LibreOffice/openpyxl 真实跑（全 mock）。

### 验证

- `.venv/bin/pytest tests/unit/ontology/ tests/integration/test_ontology_api.py -q`：**46 passed**。

### 已知权衡
- 镜像 `+约 150~200MB`（用户已确认接受）。
- `.doc` 表格内容随 python-docx paragraphs 抽取（与现有 `.docx` 行为一致，不含表格单元格）；日后需 `.doc` 表格可改转 `txt` 过滤器。
