# ToB 销售陪跑 Agent 里程碑规划

## 1. 产品定位

本项目要构建一个面向 ToB 企业客户销售团队的销售陪跑 Agent。它通过钉钉机器人作为第一入口，结合通用销售方法论和企业私有知识库，为销售人员提供情绪支持、知识查询、销售话术生成、异议处理、销售复盘和跟进建议。

第一阶段不接入 CRM，不读取客户历史跟进数据，不自动对外发送消息，不自动承诺价格、折扣、交付周期或合同条款。Agent 的定位是“内部销售辅助与训练工具”，不是替代销售对客户做承诺或执行动作的系统。

## 2. 核心原则

### 2.1 多租户优先

产品是卖给多家企业的 ToB SaaS，因此第一天就必须按多租户设计：

- 每个企业有独立 `tenant_id`。
- 每个企业的知识库独立存储、独立索引、独立检索。
- 所有 RAG 查询必须显式携带 `tenant_id`。
- 不允许只靠 prompt 约束来防止跨企业信息泄漏。
- 日志、反馈、风险记录也必须按租户隔离。

### 2.2 通用能力和企业知识分层

Agent 的能力应拆为三层：

- 通用销售能力层：销售方法论、话术结构、异议处理框架、销售复盘标准、风险规则。
- 企业私有知识层：产品资料、FAQ、案例、竞品资料、价格政策、内部销售规范。
- 渠道适配层：钉钉机器人、消息接入、身份映射、企业授权。

这样可以避免每交付一家客户都变成重度定制项目。

### 2.3 先辅助，后执行

第一版只做建议和草稿，不直接执行高风险动作：

- 可以生成客户回复草稿。
- 可以提醒销售下一步动作。
- 可以总结资料和提供销售策略。
- 不自动给客户发消息。
- 不自动承诺价格和合同条件。
- 不自动写入客户资料或修改业务系统。

### 2.4 可评估优先

Agent 不能只靠主观感觉判断效果。每个阶段都要记录：

- 用户问题
- 任务类型
- 检索到的知识来源
- 输出内容
- 风险检查结果
- 用户反馈
- 是否被复制或采纳
- 是否需要人工介入

## 3. 范围定义

### 3.1 第一版包含

- 销售情绪支持
- 企业知识库问答
- 销售话术生成
- 客户异议处理
- 销售沟通复盘
- 简单跟进建议
- 风险检查
- 对话日志
- 反馈采集
- 本地调试界面或 API
- 钉钉机器人接入

### 3.2 第一版不包含

- CRM 接入
- 客户画像自动识别
- 客户历史沟通自动读取
- 自动给客户发消息
- 自动更新商机阶段
- 自动发起审批
- 自动报价
- 自动生成合同条款
- 复杂多 Agent 协作
- 全渠道同时上线

## 4. 目标用户

### 4.1 企业销售

主要使用者，典型需求：

- 客户不回复，想要跟进话术。
- 客户嫌贵，想要异议处理。
- 不熟悉产品资料，想快速查知识库。
- 准备拜访客户，想要开场和提问框架。
- 贴一段聊天记录，想知道哪里说得不好。

### 4.2 销售主管

第二类使用者，典型需求：

- 查看销售团队常见问题。
- 发现新销售的训练短板。
- 提炼高频客户异议。
- 沉淀优秀话术。
- 审查 Agent 是否存在风险输出。

### 4.3 企业管理员

后台管理者，典型需求：

- 上传企业知识库资料。
- 配置禁用话术和风险规则。
- 查看使用数据。
- 管理企业成员和权限。

### 6.1 目标

### 6.2 工作项

- 定义第一版核心任务。
- 定义每类任务的输入、输出和失败兜底。
- 定义高风险内容范围。
- 定义企业知识库的资料类型。
- 定义租户隔离原则。
- 定义第一版指标体系。
- 准备 100-300 条测试问题。

### 6.3 输出物

- 产品定位文档
- 任务清单
- 风险规则清单
- 输出模板
- 测试样例集
- 本地 Agent spec

### 6.4 完成标准

- 明确第一版只做内部辅助，不做自动外发。
- 明确至少 5 类核心任务。
- 每类任务有标准输出格式。
- 至少有 50 条可用于本地测试的问题。

## 7. M1：本地 Agent 原型

### 7.1 目标

先不接钉钉，做一个本地可运行的 Agent API 或 CLI，用来验证核心逻辑：

- 能识别用户任务。
- 能检索企业知识库。
- 能生成销售建议和话术。
- 能做风险检查。
- 能记录对话日志。

### 7.2 推荐形态

第一版可以提供两个入口：

- 本地 HTTP API：方便后续接钉钉。
- 简单命令行或测试脚本：方便调试任务和评估。

已确认 v0 同时提供 HTTP API 和简单 CLI。本地 Web 测试页面需要做，但排在 API、CLI、RAG 和风险检查之后。

### 7.3 模块

- `TenantManager`：租户识别和配置加载。
- `KnowledgeIngestor`：资料导入、切分、元数据处理。
- `Retriever`：按 `tenant_id` 检索企业知识。
- `TaskRouter`：识别任务类型。
- `ModelProvider`：按配置调用不同 chat/embedding 模型，支持 `api_key`、`base_url` 和模型名。
- `AgentExecutor`：根据任务编排提示词和工具调用。
- `RiskChecker`：检查价格承诺、敏感外发、夸大承诺等风险。
- `ResponseFormatter`：统一输出结构。
- `ConversationLogger`：记录输入、检索、输出和反馈。

### 7.4 完成标准

- 本地可以创建至少 2 个租户。
- 每个租户可以导入不同知识库。
- 同一个问题在不同租户下只检索对应租户资料。
- 支持 5 类任务的输出。
- 风险内容能被标记或改写。
- 有基本日志。

## 8. M2：企业知识库最小闭环

### 8.1 目标

把企业资料从“文件”变成 Agent 可检索、可引用、可审计的知识库。

### 8.2 资料类型

第一版支持：

- Markdown

v0 将 Markdown 作为标准入库格式。TXT、PDF、Word、网页等资料先通过外部工具转换为 Markdown，再进入导入、切分和向量化流程。转换工具可以独立提供，不作为 v0 核心实现范围。

企业侧通常没有现成文档系统，因此本系统需要保存原文件或转换后的 Markdown、生成 chunks、写入向量数据库，并为检索命中的 chunk 提供本系统受控的片段查看能力。

### 8.3 元数据

每个文档至少保留：

- `tenant_id`
- `document_id`
- `title`
- `source_type`
- `source_uri`
- `version`
- `uploaded_by`
- `uploaded_at`
- `visibility`
- `tags`

每个 chunk 至少保留：

- `tenant_id`
- `document_id`
- `chunk_id`
- `chunk_text`
- `section_title`
- `source_file_id`
- `source_path`
- `chunk_start_line`
- `chunk_end_line`
- `page_number`
- `token_count`

### 8.4 检索要求

- 检索必须按 `tenant_id` 过滤。
- 输出企业政策、价格、案例时必须带来源。
- 销售可见来源 v0 先只展示标题。
- 结构化来源必须保留 `snippet_ref`，未来通过本系统 `snippet_url` 打开命中片段。
- 不在销售消息中展示 chunk 全文。
- 如果知识库没有依据，不能编造。
- 低置信度时要提示“需要向企业负责人确认”。

### 8.5 完成标准

- 可以上传或导入企业资料。
- 可以检索并返回引用来源。
- 可以查看某次回答用了哪些来源。
- 能发现无资料支撑的问题并拒绝编造。

## 9. M3：销售任务能力打磨

### 9.1 目标

把 Agent 从“能回答”打磨成“销售觉得有用”。

### 9.2 五类任务

#### 情绪支持

目标：销售遇到挫败、焦虑、犹豫时，先稳定情绪，再给行动建议。

输出：

- 情绪回应
- 问题判断
- 建议策略
- 推荐话术
- 下一步动作

#### 知识库问答

目标：基于企业资料回答产品、案例、方案、FAQ、竞品问题。

输出：

- 直接回答
- 依据来源
- 销售使用建议
- 风险提醒

#### 销售话术生成

目标：按销售场景生成可复制话术。

输出：

- 场景判断
- 话术版本
- 使用提醒
- 禁止表达

#### 异议处理

目标：识别客户异议类型，给出处理策略。

输出：

- 异议类型
- 客户真实顾虑假设
- 回应策略
- 推荐话术
- 追问问题

#### 销售复盘

目标：分析销售提供的沟通记录，帮助销售进步。

输出：

- 做得好的地方
- 主要问题
- 风险表达
- 更好的替代表达
- 下次训练重点

### 9.3 完成标准

- 每类任务至少有 20 条测试样例。
- 80% 以上样例能稳定输出正确格式。
- 高风险输出能被风险检查拦截。
- 销售能在 30 秒内理解并使用回复。

## 10. M4：钉钉机器人接入

### 10.1 目标

把本地 Agent 接入钉钉，让销售在真实工作入口使用。

第一版钉钉接入优先支持单聊，群聊后置。原因是单聊更容易保证上下文隔离、用户身份清晰和风险边界稳定；群聊涉及多人话题混合和上下文污染，适合在单聊稳定后再扩展。

钉钉不作为上下文管理系统。新话题、清空上下文、恢复上下文都由本系统会话层实现。v0 主路径采用文本命令，例如“新话题”“清空上下文”“重新开始”“/reset”。

### 10.2 推荐路径

验证阶段：

- 使用钉钉机器人作为消息入口。
- 后端接收消息并调用 Agent API。
- 将 Agent 回复发送回钉钉。

产品化阶段：

- 做企业应用/机器人安装和授权。
- 建立企业和 `tenant_id` 的绑定。
- 支持企业管理员配置知识库。

### 10.3 关键问题

- 钉钉回调验签。
- 企业身份映射。
- 用户身份映射。
- 群聊和单聊的上下文差异。
- 消息长度限制。
- 超时处理。
- 钉钉接口失败重试。
- 机器人被多人同时调用时的上下文隔离。

### 10.4 完成标准

- 销售可在钉钉中向机器人提问。
- 系统能识别企业租户。
- Agent 回复能在合理时间内返回。
- 日志能关联钉钉企业、用户和会话。
- 出现高风险内容时能提示销售确认，而不是直接输出危险话术。

## 11. M5：小范围企业试点

### 11.1 目标

用真实销售团队验证产品价值和失败点。

### 11.2 试点范围

建议选择 1-2 家企业，每家 5-20 名销售，试用 2-4 周。

### 11.3 试点前准备

- 企业知识库导入。
- 企业禁用表达配置。
- 销售使用说明。
- 主管反馈通道。
- 日志和问题复盘机制。

### 11.4 观察指标

短期指标：

- 日活销售数
- 人均提问次数
- 话术生成次数
- 复制/采纳率
- 点赞/点踩率
- 风险拦截次数
- 知识库无答案次数

中期指标：

- 跟进频率提升
- 沉默客户再触达率
- 新销售上手时间缩短
- 主管答疑压力下降
- 常见异议处理质量提升

长期指标：

- 商机推进率
- 成交率
- 客单价
- 销售周期

### 11.5 完成标准

- 至少有一批真实高频问题。
- 能识别 5-10 个最常见失败模式。
- 销售认为至少一个核心场景明显有用。
- 企业愿意继续试用或付费验证。

## 12. M6：SaaS 管理后台雏形

### 12.1 目标

### 12.2 后台能力

企业管理员：

- 管理企业知识库。
- 查看知识库处理状态。
- 配置风险规则。
- 查看使用统计。
- 查看高频问题。
- 查看低质量回答。

平台管理员：

- 创建租户。
- 管理租户套餐。
- 查看系统健康状态。
- 查看全局风险日志。
- 管理模型和提示词版本。

### 12.3 完成标准

- 新客户可以通过标准流程开通。
- 上传资料后可以自动进入可检索状态。
- 管理员可以看到基本使用数据。
- 可以回溯某条回答的检索来源和风险检查结果。

### 13.2 重点能力

## 15. 风险规则初稿

### 15.1 必须拦截或降级

- 自动承诺价格、折扣、返利。
- 自动承诺交付周期、实施周期、售后 SLA。
- 自动承诺法律、财务、合规责任。
- 引导销售欺骗客户。
- 生成攻击竞品、造谣竞品的话术。
- 输出企业知识库中不存在的政策。
- 泄露其他企业资料。
- 输出对客户施压、威胁、羞辱式表达。

### 15.2 应提示人工确认

- 价格相关建议。
- 合同条款相关建议。
- 大客户定制承诺。
- 交付资源安排。
- 竞品对比中的事实性断言。
- 引用客户案例中的敏感客户名称。

### 15.3 可以正常输出

- 通用销售沟通建议。
- 客户异议处理框架。
- 内部学习和复盘建议。
- 基于知识库的产品说明。
- 不涉及承诺的跟进话术草稿。

## 16. 第一版评估集设计

### 16.1 样例分类

- 情绪支持：20-50 条。
- 知识库问答：50-100 条。
- 话术生成：50-100 条。
- 异议处理：50-100 条。
- 销售复盘：20-50 条。
- 风险测试：50 条。
