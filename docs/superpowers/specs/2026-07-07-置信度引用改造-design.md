# 钉钉端置信度引用改造设计

- **日期**: 2026-07-07
- **状态**: 已与用户对齐，待写实现计划
- **作用端**: 以钉钉 stream（生产入口）为主

## 背景

钉钉端问「福多多竞品」，回答正文里每条引用都带 `(置信度80%)`，且所有来源置信度都是 80%。排查结论：这个 80% 不是真实检索分数，而是两层硬编码叠加：

1. **表面层（用户看到的文字）**：`src/sales_agent/prompts/system.py:88` 的 `SYSTEM_CONSTRAINT` 写死了一条「引用来源+置信度」规则，含「库类别 → 固定百分比」映射表（产品库 90%、策略库 85%、**竞品库 80%**、员工案例库 80%、客户画像库 75%、网络搜索 70%），命令 LLM 在正文里自报。LLM 照做。
2. **元数据层（不直接显示）**：ontology 检索的 `confidence` 在 `graph/retrieval/ontology_graph.py:232`、`graph/nodes/retrieval.py:173`、`ontology/retrieval_service.py:143` 被硬编码为 `0.8`，进 `sources.score`，但钉钉卡片从不渲染 sources。

钉钉卡片显示的是 `generate_node` 里 LLM streaming 出来的纯文本（`graph_stream.py` 的 `accumulated_text`），`state.sources` 虽然存在于 graph state（`state.py` 的 `sources: Annotated[list[dict], add]`），但从未被渲染进显示文本，也未被 `generate_node` 消费。

## 目标

1. **删掉正文自报置信度**：不再出现 `(置信度80%)` 这类文字。
2. **文末引用列表**：在回答末尾用「编号 + 文末列表」形式给出引用来源，每条格式 `[N] 标题 · 来源类型`，来源类型用「知识图谱 / 知识库 / 网络搜索」三档（代替数值置信度，让销售一眼判断这条有多靠谱）。
3. **以钉钉 stream 为主**：删自报规则是 prompt 层、全局生效；文末来源列表的渲染在钉钉 `graph_stream` 实现。

## 非目标

- **不改检索分数体系**：不补跑向量相似度、不折算分档。ontology `confidence=0.8` 硬编码保留在 metadata 不显示，本次不处理，后续可单独清理。
- **不动 HTTP `/agent/chat` 老路径、console 渲染**。
- **不做内联角标 `[1][2]`**：不依赖 LLM 在正文里配合加角标。

## 方案：代码层拼接（方案 A）

LLM 只负责生成正文；文末引用列表由代码在 finalize 时用 `state.sources` 格式化后拼接。LLM **不参与**来源列表生成，保证来源标题原样、可控、可测、零额外 token。

被否方案 B（让 LLM 自己在末尾列引用）：来源标题易被 LLM 改写/漏掉/编造，不可控。

## 详细设计

### 1. 来源类型映射

`source_type` → 中文标签：

| source_type | 标签 |
|---|---|
| `ontology` | 知识图谱 |
| `rag` / `vector` / `keyword` / `hybrid` | 知识库 |
| `web` | 网络搜索 |
| 其它 / 缺失 | 知识库（兜底） |

> 实现时需确认 `Retriever.to_source_item()` 的 `source_type` 实际取值；web 兜底 sources 目前不带 `source_type`，需在出口补 `"source_type": "web"`。

### 2. 改动点（4 处）

**P1 — `src/sales_agent/prompts/system.py:88`**
删掉「引用来源+置信度」整条自报规则（含「库类别→百分比」映射表），替换为：
> 正文中无需标注置信度与来源编号；系统会在回答末尾统一附上引用来源。

**P2 — `src/sales_agent/graph/nodes/generation.py` `generate_node`**
返回 `answer_dict` 时带上 sources，使钉钉端能从 `final_answer` 拿到：
```python
answer_dict = await execute_agent(...)
answer_dict["sources"] = state.get("sources", [])  # 新增
return {"answer_dict": answer_dict, ...}
```
`skip_generation` 分支（返回 `{}`）与无模型 fallback 分支也要保证后续 `answer_dict` 能取到 `sources`（空列表兜底）。

**P3 — `src/sales_agent/integrations/dingtalk/graph_stream.py`**
新增 `format_citation_block(sources: list[dict]) -> str`，在 `display_text` 确定后（约 line 189 之后）拼接：
```python
if display_text:
    citation = format_citation_block(final_answer.get("sources", []))
    if citation:
        display_text += citation
await card_sender.streaming_finalize(card_id, display_text)
```

**P4 — web sources 补 `source_type`**
`graph/retrieval/web_fallback.py`（或 `retrieval.py` 的 web 分支）出口给每个 source 补 `"source_type": "web"`，否则会被映射兜底成「知识库」。

### 3. `format_citation_block` 规则

输出形态：
```
──────────
📖 引用来源
[1] {title} · {type_label}
[2] {title} · {type_label}
[3] {title} · {type_label}
```
- **上限 3 条**；
- 按 `title` **去重**；
- 标题缺失 fallback：「知识库来源」（ontology/rag）/「网络来源」（web）；
- `sources` 为空 → 返回空串（不输出引用块）。

### 4. 边界处理

- **无检索**（闲聊 / script gen / skip 路径）：`sources` 空 → 正文照常、无引用块。
- **streaming**：引用块在 `streaming_finalize` 一次性拼上，不走 token streaming（避免卡片闪烁、避免与正文 token 流竞争）。
- **标题含 markdown 特殊字符**：原样输出（钉钉为 markdown 卡片）。

### 5. 测试与验证

**单元测试**：`format_citation_block` 覆盖 ontology / rag / web / 空 / 缺标题 / 去重 六种情况。

**端到端验证（必须走生产入口）**：
- `docker logs <tenant>-stream` 确认 stream 容器连上钉钉、无 crash；
- 钉钉端问「福多多竞品」，验证：① 正文不再出现任何 `(置信度NN%)`；② 末尾出现 `📖 引用来源` 编号列表，每条带「· 知识图谱」类标签。

> 依据 CLAUDE.md：HTTP `/agent/chat` 走 ChatPipeline 老路径，stream 走 graph 新路径，两条路径不同；HTTP 200 不代表 stream 健康，必须查 stream 容器日志。

## 验收标准

1. 钉钉端正文不再出现任何 `(置信度NN%)` 文字。
2. 有检索来源时，回答末尾出现编号引用列表（≤3 条，标题 + 来源类型标签）。
3. 无检索来源时（闲聊），正文末尾不出现引用块。
4. web 兜底来源标「网络搜索」。
5. stream 容器无 crash；HTTP 老路径、console 渲染不受影响。
